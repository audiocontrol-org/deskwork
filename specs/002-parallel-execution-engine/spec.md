# Feature Specification: Parallel, worktree-isolated, multi-backend execution engine

**Feature Branch**: `feature/pluggable-lifecycle-providers` (spec dir `specs/002-parallel-execution-engine`)

**Created**: 2026-06-04

**Status**: Draft

**Input**: User description: "Parallel, worktree-isolated, multi-backend EXECUTION engine for dw-lifecycle — the north-star 'execute' half (the slice after governance slice 001). deskwork takes a provider's dependency-annotated plan (concretely a Spec Kit tasks.md with its [P] markers and dependency ordering) and executes the plan's independent tasks concurrently, each isolated in its own git worktree, fanning the work across multiple pluggable execution backends, then reconciling results back. Differentiator: cross-backend execution across distinct LLM coding agents, not one vendor's subagents. CRITICAL CONSTRAINT: backends pluggable behind a capability port; the engine must NOT assume any backend's batch/headless CLI mode is available (vendors may sunset it); must support both in-session sub-agent dispatch and batch CLI shell-out, selecting/failing-over by declared capability never by vendor identity; fail loud when no backend can run. Respect the dependency graph; isolate per worktree; reconcile with conflicts surfaced; compose with slice-001 governance over a defined seam."

> **North star** (see `../../docs/1.0/001-IN-PROGRESS/pluggable-lifecycle-providers/prd.md` § North Star): deskwork as the provider-agnostic control plane that **governs** *and* **executes (parallel, multi-backend, worktree-isolated)** any provider's dependency-annotated plan. Slice 001 delivered the **governance half**. This slice delivers the **execution half** — the headline differentiator, because prior art parallelizes one model's sub-agents while nobody fans a plan across *distinct* coding agents.

> **Product home & strategy (operator decision 2026-06-04).** This feature belongs to a new plugin, **`stack-control`** (CLI binary `stackctl`; brand: stackcontrol.org), built in **this monorepo** as a new workspace package + plugin shell with its **own version line**, isolated from `dw-lifecycle`. `stack-control`'s **founding feature is slice 001** — the `deskwork-governance` Spec Kit `after_implement` extension built the prior session; **this parallel execution engine is Feature 2** (after Feature 0 plugin infrastructure + Feature 1 the `stackctl`/front-door slice), not the founding one. `stack-control` is the intended **successor to `dw-lifecycle`**: over subsequent slices it absorbs the keepers from `dw-lifecycle` (scope-discovery, audit-barrage, session-start/session-end, …) and rehomes slice 001 itself; once `stack-control` reaches parity, `dw-lifecycle` is retired. That migration is its **own multi-slice program** and is OUT OF SCOPE for this spec. Isolation rationale: `dw-lifecycle` is in active use doing real work; `stack-control` must be developed and published without destabilizing it. Consequence for the execution→governance seam (FR-017/FR-018): it crosses the plugin boundary today (`dw-lifecycle` owns audit-barrage) and is designed to remain a one-way seam when governance later moves into `stack-control`.

> **Scope: the PARALLEL engine only — re-scoped 2026-06-04 (resequence).** Execution has two modes (native Spec-Kit-with-extensions + the parallel multi-backend engine), but the 2026-06-04 resequence split them across features: **native execution rides in Feature 1** (`stackctl` + the thin control-plane front door, which triggers the native `/speckit-implement` mechanism with governance firing), and **this spec covers ONLY the parallel multi-backend engine** (the differentiator), as **Feature 2**, built *through* the Feature-1 front door. The earlier "add a native-mode user story to this body" item is therefore **dissolved** — native execution is specced under Feature 1, not here. This spec stays **paused** until the front door (Feature 1) exists.

> **Two distinct pluggability axes — keep them straight.** (1) The **provider / plan-source port** (where the plan comes from) is **deferred** in this slice: it is built **concretely against Spec Kit's `tasks.md`**; provider generalization comes in a later slice (operator decision 2026-06-04 — build against one real provider first, generalize once it works). (2) The **execution-backend port** (how each task is run — in-session sub-agent vs. batch CLI) is **fully in scope** here — it is the heart of the slice and carries the operator's batch-CLI-resilience constraint. "Build against Spec Kit first" applies to axis (1) only; axis (2) stays pluggable.

## Clarifications

### Session 2026-06-06

- Q: v1 backend roster (FR-015) → A: in-session sub-agent dispatch + **two distinct batch-CLI backends** (three backends total) — chosen so cross-backend fan-out across distinct coding agents (US3) is demonstrated, not merely asserted.
- Q: Single-task-failure disposition (FR-019) → A: v1 is conservative — **retry the failed task once**; on a second failure, record it, skip its dependents, and **continue the run** (reconciled with the unattended directive below — "halt" applies to the task's lineage, not the whole run). A richer resilience layer (configurable retry policies + per-backend circuit breakers, drawing on established microservices practice) is captured as **required future work**, explicitly NOT built in this slice (operator decision — capture, don't build; Principle II). See FR-019, FR-020.
- Q: Task → backend assignment policy (FR-010/FR-015) → A: **capability-match first, then round-robin** across the eligible backends.
- Q: Overarching operational mode → A: **fully unattended — "a system that can run all night with no operator input"** (operator directive 2026-06-06). The engine never blocks on a prompt; every decision point auto-resolves or quarantines-and-continues; a run always ends with a report. This directive governs the FR-007 and FR-019 resolutions. See FR-021, SC-010/SC-011.
- Q: Reconcile/merge policy (FR-007) → A: **isolated per-run integration branch + per-task auto-merge (no operator gate); preserved per-task branches; governance runs after, one-way non-blocking; operator promotes the run branch off the critical path.** See FR-007.
- Q: Reconcile-conflict handling refinement (operator) → A: conflicts are NOT parked for the operator by default — the engine resolves each conflict as a **dispatched task** (capability-selected backend, given both tasks' intent) and applies it. A resolver task that can't produce a resolution is a task failure (FR-019). This keeps runs fully unattended (FR-021) while guarding against bad auto-merges. See FR-007.
- Q: Reconcile-audit granularity (operator) → A: the conflict audit runs **once across the entire merge solution, and only if conflicts occurred** — NOT per conflict site. It is the one-way post-run governance pass (FR-017/FR-018) made mandatory when conflicts were resolved; findings are recorded and gate promotion, not the run. Cheaper, and it catches cross-resolution interactions a per-site audit would miss. See FR-022, SC-005, SC-012.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Execute an independent-task tranche in parallel, each isolated, then reconcile (Priority: P1) 🎯 MVP

A maintainer hands the engine a dependency-annotated plan. The engine identifies the set of tasks whose dependencies are all already satisfied (the eligible tranche), executes those tasks **concurrently** — each in its own isolated git worktree so their file writes cannot collide — and, as each completes, reconciles its result back toward the integration branch. The maintainer gets the combined result of many tasks done in the wall-clock time of roughly the slowest single task, not the sum.

**Why this priority**: This is the smallest faithful proof of the north-star execution half — parallelism + isolation + reconcile over a real plan. Everything else (backend independence, cross-backend fan-out, multi-tranche scheduling, governance) layers on top of this core loop. Without it there is no execution engine.

**Independent Test**: Feed a plan containing N tasks with no dependencies between them; observe N isolated worktrees created, the N tasks running concurrently (overlapping in time), and each task's changes reconciled into the integration branch with no cross-task file clobbering.

**Acceptance Scenarios**:

1. **Given** a plan with N mutually-independent tasks, **When** the engine runs, **Then** it creates one isolated worktree per concurrently-executing task and the tasks execute overlapping in time (not strictly sequentially).
2. **Given** N tasks executing in isolated worktrees, **When** each completes, **Then** its changes are reconciled toward the integration branch and no task's output overwrites another's silently.
3. **Given** the run completes, **When** the maintainer inspects the integration branch, **Then** it contains the combined result of all N tasks.

---

### User Story 2 - Backend independence: the run survives loss of any one backend's batch mode (Priority: P1)

The same plan must execute correctly regardless of *how* the underlying coding work is dispatched. The engine talks to execution backends only through a capability port. It supports at least two backend kinds — (1) **in-session sub-agent dispatch**, executing a task via the host Claude session's sub-agent mechanism, and (2) **batch CLI shell-out**, spawning an external LLM coding CLI headlessly — and a run completes whether both are available or only one is. Backend selection and fail-over are decided by each backend's **declared capabilities**, never by a hardcoded vendor/tool name. If a backend's batch mode becomes unavailable, eligible work routes to an available backend.

**Why this priority**: This is the operator's load-bearing constraint and the reason the engine is durable: AI-coding vendors may sunset batch/headless CLI usage (e.g. a vendor deprecating its headless print mode). An engine hardwired to one dispatch mechanism dies when that mechanism is withdrawn. Capability-based pluggability is what makes the engine outlive any single vendor's CLI surface — and it mirrors the constitution's Principle III (branch on capabilities, never provider identity).

**Independent Test**: Run the same plan three ways — both backend kinds enabled, only the in-session backend enabled, and only a batch-CLI backend enabled — and confirm the run completes in all three. Then grep the engine's backend-selection path and confirm zero branches on a vendor/tool name.

**Acceptance Scenarios**:

1. **Given** only the in-session sub-agent backend is available (every batch CLI disabled), **When** the engine runs a plan, **Then** the plan completes — proving no hard dependency on batch/headless CLI.
2. **Given** only a batch-CLI backend is available, **When** the engine runs the same plan, **Then** the plan completes.
3. **Given** a task is assigned to a backend whose batch mode has become unavailable, **When** the engine schedules it, **Then** the task routes to another backend that declares the required capability, without operator intervention.
4. **Given** the engine selects a backend for a task, **When** its selection logic is inspected, **Then** the decision is made from declared capabilities only, with no branch on a vendor/tool identity string.

---

### User Story 3 - Cross-backend fan-out across distinct coding agents (Priority: P2)

In a single run, the engine distributes the eligible tasks across **two or more distinct** execution backends simultaneously — e.g. some tasks executed by one coding agent and others by a different one — rather than confining all parallelism to a single vendor's sub-agents.

**Why this priority**: This is the headline differentiator versus prior art (MAQA, Fleet, single-model sub-agent parallelism). It is a strict superset of US1's parallelism and depends on the US2 backend port existing, so it follows them.

**Independent Test**: Enable two distinct backends, run a plan whose tranche has at least two independent tasks, and confirm the run record attributes different tasks to different backends within the same execution.

**Acceptance Scenarios**:

1. **Given** at least two distinct backends are enabled and a tranche has ≥2 independent tasks, **When** the engine runs, **Then** the run distributes tasks across ≥2 distinct backends concurrently.
2. **Given** a completed cross-backend run, **When** the maintainer inspects the run record, **Then** it shows which backend executed each task.

---

### User Story 4 - Dependency-respecting multi-tranche scheduling (Priority: P2)

For a plan with dependency edges, the engine sequences work as a series of tranches: a task becomes eligible only once **all** its dependencies have completed; the engine runs each eligible tranche in parallel, then recomputes eligibility and runs the next tranche, until the plan is exhausted.

**Why this priority**: Real plans (a Spec Kit `tasks.md`) have dependency ordering, not just a flat independent set. US1 proves a single tranche; this proves correct sequencing across the whole graph. It builds directly on US1.

**Independent Test**: Feed a plan with a known dependency chain (A → B, with C independent); confirm B never starts before A completes, while C may run alongside A.

**Acceptance Scenarios**:

1. **Given** task B depends on task A, **When** the engine schedules, **Then** B does not start until A has completed.
2. **Given** task C has no dependencies, **When** the first tranche runs, **Then** C is eligible in that tranche alongside other dependency-free tasks.
3. **Given** a plan whose remaining tasks all still have unmet dependencies and none are in progress, **When** the engine evaluates eligibility, **Then** it halts with a descriptive error rather than looping or silently completing (a dependency cycle / unsatisfiable plan is surfaced, not swallowed).

---

### User Story 5 - Execution → governance seam (Priority: P3)

When a run (or tranche) completes, the engine produces a well-defined handoff that slice-001 governance consumes to audit the produced work — without manual glue. The engine does not re-implement governance; it exposes the seam governance needs (which work was produced, where) so the existing cross-model audit-barrage can run over it.

**Why this priority**: The north star is *govern + execute*. Governance already exists (slice 001); this slice must compose with it rather than duplicate it. The seam is the integration point, but it depends on a completed run existing first, so it is last.

**Independent Test**: Run the engine to completion over a plan, then confirm slice-001 governance can be invoked over the run's produced work using only the handoff the engine emits (no hand-assembled inputs).

**Acceptance Scenarios**:

1. **Given** a completed engine run, **When** governance is invoked, **Then** it audits the run's produced work using only the engine's emitted handoff.
2. **Given** the seam, **When** it is inspected, **Then** governance consumes execution output one-way and the engine never depends on governance internals (clean, one-directional seam).

---

### Edge Cases

- **No backend can run an eligible task** (every backend declaring the required capability is unavailable): the engine fails loudly with a descriptive error naming the missing capability — it does NOT silently skip the task or fall back to mock output (Principle V).
- **Reconcile conflict** between two parallel tasks' changes: the engine dispatches the conflict resolution as a task to a capable backend (given both tasks' intent) and applies it; if a resolver task can't produce a resolution, that's a task failure (FR-019: retry once, then quarantine, continue). After assembly, **iff any conflict occurred (resolved or quarantined)**, a single whole-solution audit-barrage runs over the run branch (FR-022); blocking findings mark the run `completed-promotion-blocked`, gating promotion (FR-030). The engine never silently picks a winner, never halts the run, and never waits for the operator (FR-007 / FR-022 / FR-021).
- **Empty eligible tranche but unfinished plan** (dependency cycle or unsatisfiable edge): the run **terminates with a descriptive unsatisfiable-plan error report** (a normal FR-004/FR-021 terminal state — `failed-before-dispatch` when detected at preflight per FR-001a, else terminate-and-report preserving all completed/merged work), never an infinite wait or a false "done."
- **A single task fails inside its backend**: the engine retries once (FR-024 defines "fails"); on a second failure it records the failure, does not start that task's dependents, and continues independent work (FR-019) — the run is not halted and does not wait for the operator (FR-021).
- **A backend dies mid-task** (vs. the task running to a failed result): **rerouted** to another capable backend without consuming the retry budget (FR-025); partial work in the dead worktree is discarded.
- **Worktree creation fails** (dirty tree, disk, name collision): surfaced as an error before dispatch; no partial silent execution. Resource exhaustion *during* a run is handled by FR-027; run-branch name collision by FR-026.
- **Provider's source plan artifact**: never written to by the engine (one-way authority, Principle IV) — progress/state lives in deskwork's substrate, not the provider's `tasks.md`.
- **The feature branch advances during a multi-hour run**: the run branch is pinned to its recorded base SHA; promotion verifies the tip or rebases against it, surfacing divergence (FR-026) — never a silent replay onto a moved base.

## Requirements *(mandatory)*

### Functional Requirements

**Plan intake & scheduling**

- **FR-001**: The engine MUST read the **Spec Kit `tasks.md`** for the active feature (its `[P]` parallel markers + the "Dependencies" section) and determine, at each scheduling step, the set of tasks whose dependencies are all satisfied (the eligible tranche). **This slice is built concretely against Spec Kit** — generalizing the plan source to other providers (a normalized manifest / provider port) is explicitly deferred to a later slice. *(Operator decision 2026-06-04: build it against one real provider first so we know we're building something that actually works, then generalize.)*
- **FR-001a** *(plan validation; barrage codex-10)*: Before dispatching any task, the engine MUST validate the parsed plan and **reject a malformed plan with a descriptive error, creating no worktrees**: duplicate or missing task ids, a dependency reference to an absent task id, and dependency edges that form a cycle are all rejected up front (fail-loud, Principle V). A plan that parses but yields an invalid graph is a defect to surface, not to schedule.
- **FR-002**: The engine MUST execute the tasks in an eligible tranche concurrently, up to a **bounded maximum concurrency** (realized as per-stage worker pools + bounded queues with backpressure — FR-032; the concrete bound/formula is a `/speckit-plan` detail).
- **FR-003**: The engine MUST recompute eligibility after each completion and continue scheduling, respecting all dependency edges. **A dependency is "satisfied" for eligibility only once it is MERGED into the run integration branch** (FR-023), never merely executed. The run reaches its single terminal condition when **no task is in progress and no task is eligible** (FR-032); it then terminates with a report (FR-021). *(There is no "until all tasks complete" loop — failed/quarantined tasks never become complete; the terminal condition is queue-drain + no-eligible, matching FR-019/FR-021.)*
- **FR-004**: The engine MUST detect an **unsatisfiable state** (non-empty remaining plan, empty eligible set, nothing in progress — a dependency cycle or an unmet edge) and **terminate the run with a descriptive unsatisfiable-plan error report** — a normal FR-021 terminal state, NOT a hang and NOT a false "done." Already-completed/merged task results are preserved and reported; unreachable tasks are recorded as blocked. *(This is "terminate with an error report," not the FR-021-forbidden "halt and wait" — the run still ends without operator input.)*

**Isolation & reconcile**

- **FR-005**: The engine MUST execute each concurrently-running task in its own isolated git worktree so parallel tasks' file writes cannot collide.
- **FR-006**: The engine MUST reconcile each completed task's changes back toward the integration branch.
- **FR-007**: On a reconcile conflict, the engine MUST NOT silently choose a winner and MUST NOT halt the run. **Reconcile policy (resolved 2026-06-06 — unattended operation; conflict-as-task refinement):** each task runs on its own **preserved per-task branch**; the engine creates an **isolated per-run integration branch** off the feature-branch tip and **sequentially auto-merges** completed task branches into it, with **no operator gate**. When a merge conflicts, the engine does NOT park it for the operator by default — it **dispatches the conflict resolution as a first-class task** through the execution-backend port (a capability-selected backend; **when the orchestrating agent resolves, it does so as a registered backend selected via the same capability filter — no privileged out-of-band path**, FR-009), giving the resolver **both conflicting tasks' intent** (from the plan) plus the conflicted merge state so it resolves *semantically*, not just textually, then applies the resolution to the run integration branch. The engine does NOT audit each resolution individually; instead, **iff one or more conflicts OCCURRED during assembly (whether resolved or quarantined), it runs a single whole-solution audit-barrage over the entire assembled run branch at the end** (FR-022) — a zero-conflict assembly needs none. If a resolver task itself fails to produce a resolution, that is a task failure under FR-019 (retry once, then quarantine that task's branch, skip dependents, continue). The engine never silently chooses a winner (SC-005); it never halts the run and never waits for operator input (FR-021). The resolution diffs are recorded in the run record. The whole-solution audit is the one-way post-run governance pass (FR-017/FR-018) made mandatory when conflicts occurred; its findings are recorded, not consumed to gate individual merges. The operator promotes the completed run branch into the feature branch at their discretion; **promotion is never on the run's critical path**, and blocking whole-solution audit findings flag the branch for review before promotion.
- **FR-008**: The engine MUST NOT write to the provider's source plan artifact; execution progress/state is recorded in deskwork's own substrate, one-way (Principle IV).
- **FR-023** *(completion = merged; barrage claude-05/claude-06)*: A task is **"complete" only once its per-task branch is MERGED into the run integration branch**, never merely when its backend finishes executing. Consequences: (a) a dependent task's worktree is created from the **run integration branch state that includes all of its dependencies' merged results**, so it sees their work; (b) a task's dependents become eligible only after that task is merged; (c) therefore a task whose merge or resolution fails (quarantined) can never have started dependents — the "dependents built on quarantined work" hole is structurally closed. Merges are serialized (the reconcile stage is single-consumer, FR-032) and happen incrementally as tasks finish.
- **FR-026** *(run integration branch lifecycle; barrage claude-18/codex-11/codex-12)*: The engine MUST name the per-run integration branch and per-task branches with a **collision-proof unique run id** (passed in — note `Date.now()`/randomness are unavailable in some execution contexts) and **refuse to start (no mutation) if an expected-new ref already exists**. It MUST **record the exact base commit SHA** the run branch was cut from; **promotion** MUST verify the feature branch still points at that base (or perform an explicit rebase/reconcile against the current tip), surfacing any divergence in the run record — the run never silently replays stale work onto a moved base.
- **FR-027** *(worktree lifecycle + resource exhaustion; barrage claude-13/codex-14/codex-15)*: The engine MUST **tear down each task's worktree directory after reconcile** (preserving only the per-task branch ref) and define a retention/GC expectation for run integration branches. **Disk/resource exhaustion during a run** (not only at pre-dispatch worktree creation) MUST be handled deterministically per FR-021 — quarantine-and-continue where the affected task can be isolated, else terminate with a descriptive resource-exhaustion error report; never a silent skip or an unbounded accumulation.

**Backend port & capabilities (the durability constraint)**

- **FR-009**: The engine MUST talk to execution backends only through a capability port; it MUST NOT branch on a vendor/tool identity anywhere in backend selection or dispatch (Principle III).
- **FR-010**: Each backend MUST declare its capabilities (at minimum: which dispatch mechanism it provides and whether it is currently available); the engine MUST select and fail-over among backends using these declarations only. **Assignment policy: capability-match first (filter to backends declaring the required capability), then round-robin across the eligible set** — fair, fully capability-driven (Principle III), and naturally produces cross-backend fan-out (US3). *(The concrete capability **vocabulary** — which capabilities exist beyond dispatch-mechanism + availability, and what each task requires — is **DEFERRED**: insufficient information to define it well without concrete backends to derive it from; resolve it at `/speckit-plan`/implementation, integration-first per Principle II. The capability-**port property** asserted here — select on declared capability, never on identity — holds regardless of the vocabulary's eventual content. See Assumptions.)*
- **FR-011**: The engine MUST support at least two backend kinds: (a) **in-session sub-agent dispatch** via the host session's sub-agent mechanism, and (b) **batch CLI shell-out** to an external coding CLI.
- **FR-012**: The engine MUST run a plan to completion when only the in-session backend is available AND when only a batch-CLI backend is available — i.e. it MUST NOT hard-depend on any batch/headless CLI being present.
- **FR-013**: When a backend assigned a task becomes unavailable (e.g. its batch mode is withdrawn), the engine MUST route the task to another backend that declares the required capability.
- **FR-014**: When no available backend declares the capability an eligible task requires, the engine MUST fail loudly with a descriptive error naming the missing capability — no silent skip, no mock output (Principle V).
- **FR-015**: In a single run, the engine MUST be able to distribute tasks across two or more distinct backends concurrently (cross-backend fan-out). **Roster — runtime requirement:** the engine MUST implement at least **one in-session adapter + one batch-CLI adapter** (the two backend *kinds*, FR-011); a run completes with whatever backends are actually installed/available (Principle IX — no hard dependency on any specific vendor's batch CLI). **Roster — conformance/demonstration requirement:** the cross-backend, distinct-coding-agent differentiator (US3) MUST be PROVEN with **≥2 distinct batch-CLI backends in the test/conformance environment** — a fixture, NOT a runtime hard-dependency *(operator decision 2026-06-06, revised from "two batch CLIs required at runtime" to protect the durability principle; barrage codex-08)*. The engine selects among all available backends by declared capability only (Principle III), never by identity.
- **FR-025** *(backend loss mid-task — precedence over FR-019; barrage claude-14/codex-09)*: **Loss of a backend before the task it is running completes is a REROUTE, not a task failure** — the engine re-dispatches the task to another capable, available backend (FR-013) **without consuming the FR-019 retry budget**; partial work in the dead backend's worktree is discarded. FR-019's retry-once applies to a task that *ran to a result that was a failure*, not to a backend that died mid-flight. *(Resolves the FR-013/FR-019 precedence the Assumptions previously punted; the reroute count may itself be bounded — a `/speckit-plan` detail.)*

**Run record & governance seam**

- **FR-016**: The engine MUST record a run artifact capturing, per task: which backend executed it, success/failure, and where its output landed.
- **FR-017**: On run/tranche completion, the engine MUST emit a handoff that slice-001 governance can consume to audit the produced work, without manual assembly of governance inputs.
- **FR-018**: The execution→governance seam MUST be one-directional (governance reads execution output; the engine does not depend on governance internals).

**Failure semantics**

- **FR-019**: When an individual task fails inside its backend, the engine MUST **retry it once**; if the retry also fails, the engine MUST record the failure against that task in the run artifact, **NOT start the failed task's dependents**, and **continue executing independent tasks** — the run proceeds unattended and terminates (with a report) only when no eligible work remains. *(Reconciled 2026-06-06 with the unattended-operation directive, FR-021: the operator's "retry once, then halt" applies to the failed task's **lineage** — its dependents do not start — not to the whole run; a single double-failure must not end an overnight run. The richer retry/circuit-breaker policy remains captured-and-deferred, FR-020.)*
- **FR-024** *(what "fails" means; barrage claude-04)*: A task's per-task outcome is judged by a **cheap liveness signal**, not semantic correctness: a task **succeeds** iff its backend **terminates non-error AND produced a non-empty committed diff** (and, if the task declares an acceptance command, that command also passes). Anything else is a failure (→ FR-019). **Correctness is NOT a per-task gate** — it is the job of the whole-solution audit (FR-022) and the post-run governance pass (FR-017). *(Per the thesis: detection over instruction — cheap per-task liveness + stochastic-correctness audit at the end. Principle I means many tasks are "write failing test → implement," so "the task's tests pass" is not a universal per-task success signal.)*
- **FR-020** *(captured, deferred — NOT built in this slice)*: A richer execution-resilience layer is **required future work**: configurable retry policies (backoff, max-attempts) and per-backend **circuit breakers**, drawing on established microservices practice (where there is substantial prior art). v1 deliberately ships only the minimal retry-once-then-continue of FR-019; the fuller policy is its own future slice. Captured per operator decision (Principle II — capture, don't build); recorded here so the need is not lost, with no v1 implementation obligation.

**Unattended operation (the load-bearing operator directive)**

- **FR-021**: The engine MUST run a plan to completion **fully unattended** — it MUST NOT block on, or wait for, operator input at any point during a run. Every decision point either resolves automatically and safely or **quarantines** the affected task (preserving its artifacts and recording it for later review) and continues. A run ALWAYS terminates with a written report; it NEVER pauses awaiting a prompt, a merge approval, or a conflict resolution. *(Operator directive 2026-06-06: "a system that can run all night with no operator input." This governs FR-007 reconcile-conflict handling — auto-resolve-then-audit, quarantine only as last resort — and FR-019 task-failure handling — retry-once then continue.)*
- **FR-022**: The reconcile-conflict audit MUST run **exactly once per run, over the ENTIRE assembled run integration branch** (never per conflict site), and **only when one or more conflicts OCCURRED during assembly (whether resolved or quarantined)** — a zero-conflict assembly runs no conflict audit; a run where a conflict could NOT be resolved still gets audited *(barrage claude-08 — the riskiest run must not be the one that escapes the only safety pass)*. It uses **cross-model audit-barrage** (slice-001 governance, applied concretely; Principle II — not abstracted into a validator port until a second validator exists). Its findings are **recorded** in the run record and the normal finding machinery; a run whose whole-solution audit surfaces blocking findings still completes (unattended, FR-021) but ends in terminal state `completed-promotion-blocked` (FR-031) and the promotion gate (FR-030) refuses it without a logged override. *(Honest seam framing — barrage claude-07/codex-17: this is stack-control's **execute half consuming its own govern half** (audit-barrage rehomes into stack-control, roadmap Feature 4) — **in-house composition**, not a foreign cross-plugin dependency, so it does not threaten the isolation invariant. It IS a runtime dependency on the **audit capability being available**; if that capability is absent the engine **fails loud naming it** (Principle V), never silently skips. The separate POST-RUN governance seam over produced work, FR-017/FR-018, stays strictly one-way. Auditing the assembled whole also catches cross-resolution interactions a per-site audit would miss, at a fraction of the cost.)*

**Durability, run terminal states & promotion**

- **FR-028** *(durability — push as produced; C4 / barrage codex-25)*: Per-task branches and the run integration branch MUST be **pushed to the remote as work is produced**, so an interrupted overnight run (machine death, crash) loses no completed work (Principle VII + the all-night directive). *(v1 default policy; requires a remote with write access during the run. Whether to also offer a local-only mode is a `/speckit-plan` detail; push-as-produced is the default.)*
- **FR-031** *(run terminal states; barrage codex-03/codex-19/claude-11)*: A run MUST end in exactly **one explicit terminal state**, surfaced in the run record AND the process exit code: `completed-clean` (all tasks merged; no quarantines; no blocking audit findings), `completed-with-quarantines` (the run finished but some tasks failed/were blocked), `completed-promotion-blocked` (whole-solution audit surfaced blocking findings — FR-022), or `failed-before-dispatch` (invalid plan / preflight failure — FR-001a/FR-026). The run report MUST surface the **quarantine ratio** so a degenerate run is visibly distinct from a clean one.
- **FR-030** *(promotion gate — mechanical, not advisory; C3 / barrage claude-11)*: Promotion of a run integration branch into the feature branch MUST be a **machine-checkable gate**: when the run is `completed-promotion-blocked`, the promotion path (`stackctl promote`) MUST **refuse** without an explicit `--override`, and the override MUST be **recorded** in the run record. *(Per the thesis — make failure states mechanically impossible, not merely discouraged; an advisory flag lets a hurried operator or a CI step ship a known-bad merge.)*

**Architecture: staged event-driven (SEDA)**

- **FR-032** *(architecture — staged, queue-based concurrency in the SEDA tradition; operator 2026-06-06)*: The execution engine MUST be realized as a **staged, queue-based concurrency pipeline** — stages (`parse → schedule → execute → reconcile → audit`) connected by **bounded queues**, each stage with its own **worker pool** and **admission control / backpressure**. **A queue-based mechanism is settled and non-optional** (operator: *"there is no option but to use some kind of queuing mechanism"*); what is deliberately left open is the *precise* realization (see the research proviso below). It carries these **behavioral requirements** (these hold regardless of the concrete mechanism the research selects): (a) **bounded concurrency with backpressure** — under load the engine throttles admission rather than exhausting resources (this is how FR-002's bound is realized); (b) the **execute stage is multi-worker** (parallelism) while the **reconcile stage is single-consumer** (merges serialize naturally — FR-007/FR-023); (c) the **single termination invariant** — the run ends when all stage queues are drained, no worker is busy, and no task is eligible (the one terminal condition FR-003/FR-019/FR-021 all reference); (d) **stage observability** — queue depths and per-stage status surfaced in the run record. *(SEDA is apt precisely because of the unattended/all-night requirement: it exists for well-conditioned behavior under overload.)*
- **FR-032a** *(architecture — state-of-the-art research proviso; operator 2026-06-06)*: Before the concrete design in FR-032 is fixed, `/speckit-plan`'s `research.md` MUST include a **state-of-the-art survey** of staged / queue-based concurrent-execution mechanisms — classic SEDA *and* its modern descendants (e.g. work-stealing pools, actor / CSP runtimes, structured concurrency, durable / orchestration execution engines) — and **select or refine the concrete realization from that survey** with a recorded rationale. The survey MAY refine the stage set, queue semantics, pool model, and admission policy; it MUST preserve behavioral requirements (a)–(d) of FR-032 and MUST NOT eliminate the queue. *(Grounds the architecture in current practice rather than a ~20-year-old paper by name; consistent with the thesis — invest heavily up front so execution can be industrialized.)*

### Key Entities *(include if feature involves data)*

- **Plan**: the provider's dependency-annotated unit of work (first concrete instance: a Spec Kit `tasks.md`). Authoritative for INTENT; never mutated by the engine.
- **Task**: a single unit within a plan, with an id, a description/intent, and zero or more dependency references.
- **Dependency edge**: a "task X requires task Y complete" relation; defines tranche eligibility.
- **Tranche**: the set of currently-eligible (all-dependencies-satisfied) tasks scheduled to run concurrently.
- **Execution backend**: a pluggable adapter that runs a task. Two kinds in scope: in-session sub-agent dispatch; batch CLI shell-out.
- **Backend capability**: a declared property of a backend (dispatch mechanism, current availability, …) — the only thing the engine branches on (Principle III). *(The concrete capability vocabulary is deferred to plan/implementation, integration-first — FR-010 / Assumptions.)*
- **Worktree**: an isolated git working tree in which a single concurrent task executes.
- **Per-task branch**: the branch a task's worktree commits to — **preserved** after the run as a durable artifact (audit trail, rollback point, the concrete "where output landed" the run record references).
- **Run integration branch**: an isolated branch created per run off the feature-branch tip, into which completed per-task branches are auto-merged; the assembled run result the operator later promotes (FR-007).
- **Reconcile/integration result**: the outcome of merging a task's per-task branch into the run integration branch (**clean-merged**; **auto-resolved-then-merged** via a dispatched resolution task; or **quarantined-after-failed-resolution** — the resolver could not produce a resolution, FR-019). *(The whole-solution audit is separate and run-level, NOT a per-merge gate — see its own entity.)*
- **Conflict-resolution task**: a first-class task whose input is a conflicted merge (base + both sides) plus both conflicting tasks' intent, and whose output is an applied resolution; dispatched through the capability port like any other task (FR-007).
- **Whole-solution reconcile audit**: a single cross-model audit-barrage over the entire assembled run branch, run once per run and only when ≥1 conflict was resolved; its findings are recorded and gate promotion, not the run (FR-022).
- **Run record**: the per-run artifact (per-task backend attribution, status incl. retried/failed/quarantined, conflict-resolution diffs, the whole-solution audit verdict when run, the recorded base SHA, output location / branch ref, the terminal state, the quarantine ratio, and any logged promotion override).
- **Run terminal state**: the single explicit state a run ends in — `completed-clean` / `completed-with-quarantines` / `completed-promotion-blocked` / `failed-before-dispatch` (FR-031); surfaced in the run record and the exit code.
- **Governance handoff**: the one-way seam describing the produced work for slice-001 governance to audit.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For K mutually-independent tasks each with measured solo duration ≥ T, the run record shows **≥2 tasks with overlapping execution intervals** (their start/end timestamps overlap) — the falsifiable evidence of real parallelism. *(Overlap, not a wall-clock ratio: for small/fast tasks, worktree + dispatch overhead can dominate the sum without disproving concurrency; barrage claude-23/codex-18/codex-26.)*
- **SC-002**: The same plan reaches terminal state **`completed-clean`** (FR-031) in **all three** backend configurations — both kinds enabled, in-session only, and batch-CLI only — proving no hard dependency on batch/headless CLI availability. *("Completes successfully" = `completed-clean`, not merely "terminated"; barrage codex-19.)*
- **SC-003**: **Given ≥2 capability-eligible backends and a tranche of ≥2 independent tasks**, a single run distributes them across **≥2 distinct backends** concurrently, evidenced by per-task backend attribution in the run record. *(Conditioned on its premises — a width-1 tranche or a single eligible backend legitimately won't fan out; barrage claude-22.)*
- **SC-004**: The backend-selection function reads **only the declared-capability struct** — verified by a **behavioral test**: swapping two backends' identities (capabilities held fixed) yields identical selection. *(A behavioral invariant, not a grep — the two named batch CLIs of FR-015 must appear somewhere, so "grep finds zero vendor strings" is false by construction; barrage claude-15. Holds regardless of the deferred capability vocabulary.)*
- **SC-005**: For a run in which conflicts occur, the engine resolves each conflict (as a dispatched task) and runs **exactly one** whole-solution audit-barrage over the assembled result — **0** silent overwrites, **0** per-site audits. A zero-conflict run runs no conflict audit. Blocking whole-solution findings flag the run branch for operator review before promotion; they never silently merge and never halt the run.
- **SC-006**: When no available backend declares an eligible task's required capability, the engine fails with a descriptive error naming the missing capability — **0** silent skips and **0** mock outputs.
- **SC-007**: The provider's source plan artifact is **byte-unchanged** before vs. after a run (one-way authority).
- **SC-008**: A completed run produces a governance handoff that slice-001 governance consumes with **no** manually-assembled inputs.
- **SC-009**: For a plan with a dependency chain, **100%** of dependent tasks start only after their dependencies complete (no early starts observed across the test plan).
- **SC-010**: A run started with no terminal/operator attached completes with **0** prompts and **0** blocking waits — every reconcile conflict is **auto-resolved-or-quarantined** (FR-007, never parked for the operator by default) and every retry-exhausted task failure is quarantined and recorded; the run terminates in an explicit terminal state with a written report rather than waiting for input (FR-021/FR-031). *(Fixed: conflicts are resolved, not quarantined-by-default; barrage claude-02/codex-21.)*
- **SC-011**: After a run, the feature/integration branch the operator works from is **unchanged** unless the operator explicitly promotes the run branch — the engine assembles onto an isolated per-run integration branch, so an unattended run never mutates the operator's working branch on its own (FR-007).
- **SC-012**: In an unattended run, the engine resolves conflicts **autonomously** (dispatched resolution tasks) and validates the assembled solution with a **single** audit pass — **0** operator interactions during the run. The operator's only touchpoint is at promotion, and only when the whole-solution audit flagged blocking findings (FR-007/FR-022).
- **SC-013** *(productivity floor; barrage claude-10)*: For a plan of N independent, individually-passing tasks with all backends healthy, the engine merges **≥ N−q** tasks into the run integration branch (q = the count of genuinely-failing tasks) — an unattended run must actually *produce*, not merely *not-block*. A degenerate (e.g. all-quarantine) run is reported via its quarantine ratio and terminal state, **never** as `completed-clean`.
- **SC-014** *(promotion gate; C3 / barrage claude-11)*: When a run is `completed-promotion-blocked`, `stackctl promote` **refuses** without an explicit `--override`, and any override is recorded — **0** silent promotions of a known-bad merge.
- **SC-015** *(unambiguous termination; barrage codex-03/claude-17)*: Every run ends in exactly one FR-031 terminal state, surfaced in both the run record and the exit code — **0** ambiguity between a clean run, a degraded run, and a preflight failure.

## Assumptions

These are reasonable defaults chosen where the description did not fix a detail. They are **starting positions to confirm in `/speckit-clarify`**, not scope cuts (Constitution Principle II — capture, then scope):

- **Concurrency bound** (FR-002): a host-derived bound (e.g. roughly available cores minus a headroom margin, capped by tranche width), **realized as SEDA per-stage pools + bounded queues with backpressure** (FR-032). *Deferred to `/speckit-plan`* — the exact bound/formula is a planning detail; the default is low-risk.
- **Task → backend assignment policy** (FR-010/FR-015): **RESOLVED (2026-06-06)** — capability-match first, then round-robin across eligible backends. See FR-010.
- **Single-task-failure disposition** (FR-019): **RESOLVED (2026-06-06)** — retry the failed task once; on a second failure record it, skip the failed task's dependents (its lineage), and **continue the run** (NOT halt — reconciled with the unattended directive, FR-021). See FR-019/FR-024; richer retry/circuit-breaker policy captured-and-deferred in FR-020.
- **Capability vocabulary** (FR-010/FR-014): **DEFERRED (2026-06-06)** — insufficient information to define the capability vocabulary content without concrete backends; derive it integration-first at `/speckit-plan`/implementation (Principle II). The capability-port *property* (select on declared capability, never identity) is fixed now; only the vocabulary content is open.
- **In-session backend concurrency** (barrage claude-16): whether the host session can run multiple sub-agents *concurrently* (vs. serialized) is a **declared backend capability** (part of the deferred vocabulary, FR-010). SC-001's overlap is demonstrated in any configuration containing a concurrency-capable backend; SC-002 requires only *completion* in in-session-only mode, not overlap — so the two are not in tension. The concrete in-session concurrency model is a `/speckit-plan` detail.
- **Plan source is Spec Kit `tasks.md`, concretely** (operator decision 2026-06-04 — no provider abstraction in this slice; generalize after it demonstrably works). The engine's internal task/dependency model is a private engine detail, NOT a published provider port.
- **Integration target** is the feature's integration branch in the same repository; worktrees are git worktrees under that repo.
- **Governance** (slice 001) already exists and is the consumer of the execution→governance seam; this slice defines the seam and does not modify governance internals.
- **Mid-run backend-loss disposition**: **RESOLVED (2026-06-06)** — a backend dying mid-task is a **reroute** without consuming the retry budget; partial work discarded (FR-025). Not-yet-started eligible work routes to available backends (FR-013).
- **Dual driver (captured, not built up front — Principle II; barrage claude-20).** The run record and governance handoff are persisted as **readable on-disk artifacts** — which the unattended report (FR-016/FR-021) requires *anyway*, so this rests on a present need, not a speculative one. Designing a *second-consumer interface* for a future `stack-control` control-plane frontend is **deferred** until that frontend is a concrete instance (Principle II — generalize after a second instance, not before). The frontend itself is a separate future feature, out of scope here. *(Reframed from a binding forward-interface mandate: there is exactly one concrete consumer today, `stackctl`.)*
