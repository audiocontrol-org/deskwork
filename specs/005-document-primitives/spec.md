# Feature Specification: Generalized document-handling primitives — archive & curate (`design/document-primitives`)

**Feature Branch**: `feature/stack-control` (authored alongside specs 001–004 on the shared branch; no dedicated branch — operator decision 2026-06-07)

**Created**: 2026-06-07

**Status**: Draft

**Input**: User description: "Reform roadmap discipline, starting with generalized document-handling primitives — `archive` and `curate` — built as part of the stack-control plugin (not project-specific tooling) and dogfooded in the project. `archive` moves non-live items out of a document into an archive, based on the *mechanism* of the dw-lifecycle workplan-archive tooling but generalized. `curate` ensures a live document is up-to-date, well-formed, well-ordered, and properly archived. Documents are self-describing via a declared grammar; in-document identifiers must be human-readable and must NOT bake in ordinal components."

## User Scenarios & Testing *(mandatory)*

The "users" are operators and agents working on living documents (roadmaps, design inboxes, specs) inside a project that has adopted the stack-control plugin.

### User Story 1 - Keep a live document lean by archiving settled items (Priority: P1)

An operator has a living document whose settled items (shipped roadmap rows, promoted/dropped inbox entries) have accumulated and are crowding the live surface. They run a single command that moves every settled item out of the live document into a sibling archive, leaving the live document containing only active items — and they can reverse it.

**Why this priority**: This is the headline value and the smallest shippable slice. A lean live document is the immediate, visible payoff; archiving is the mechanism the operator reached for first ("an archive skill to keep live documents lean").

**Independent Test**: Take a governable document with a mix of active and settled items, run `archive --apply`, and confirm (a) every settled item now lives in the sibling archive, (b) the live document retains exactly the active items, (c) the move is recorded in a provenance ledger, and (d) `unarchive --apply` returns the unit to its declared-order position relative to the live document's current neighbors (touching only that unit), leaving the document content-equivalent — and well-ordered when the live document was already well-ordered before the round-trip.

**Acceptance Scenarios**:

1. **Given** a governable document with three active items and two settled (terminal-status) items, **When** the operator runs `archive --apply`, **Then** the two settled items are removed from the live document and appended to the sibling archive file, and the ledger records both moves keyed by their identifiers.
2. **Given** the same document, **When** the operator runs `archive` without `--apply`, **Then** the planned moves are reported and nothing is written (dry-run is the default).
3. **Given** an archived item, **When** the operator runs `unarchive --apply` for that item, **Then** the item returns to the live document at its declared-order position **relative to the live document's current neighbors** (only that item is touched; the rest is not reordered) and the ledger entry is removed, leaving the document content-equivalent — and well-ordered when the live document was already well-ordered before the round-trip.
4. **Given** an archive run, **When** it completes, **Then** the provenance ledger matches the archive file's actual contents (coherence holds).

### User Story 2 - Curate a live document so it stays correct (Priority: P2)

An operator wants assurance that a living document is structurally sound: it conforms to its declared shape, its items are in the intended order, settled items have been archived, and — when the document opts in — it is reconciled against an external source of truth. They run one command that reports every deviation and, on request, fixes the mechanical ones.

**Why this priority**: Curation is the broader "ensure it's right" capability that composes archiving. It is higher-effort than archive alone and depends on the same engine, so it follows the MVP.

**Independent Test**: Take a governable document that is out of order, has an un-archived settled item, and declares a reconciliation source; run `curate`; confirm it reports the disorder, the missing archival, and the reconciliation hook as *recognized but not executed*; run `curate --apply` and confirm the items are reordered AND the settled item is archived.

**Acceptance Scenarios**:

1. **Given** a governable document whose items are out of the declared order, **When** the operator runs `curate --apply`, **Then** the items are reordered to the declared order key (and any settled items are then archived), and no item identity changes.
2. **Given** a document containing a settled item that has not been archived, **When** the operator runs `curate`, **Then** the report flags it as belonging in the archive; **When** the operator runs `curate --apply`, **Then** the settled item is moved to the archive (curate composes `archive --apply`).
3. **Given** a document that does not parse against its grammar, **When** the operator runs `curate`, **Then** curate fails loud and names the offending location; it does not attempt a partial fix.
4. **Given** a document whose grammar declares a reconciliation hook, **When** the operator runs `curate`, **Then** curate reports the hook as "declared, not executed" (execution is out of scope for this feature — a separate feature owns it) and the other checks still run.
5. **Given** a document whose grammar declares no reconciliation hook, **When** the operator runs `curate`, **Then** the up-to-date check is silent and the other checks still run.

### User Story 3 - Identities stay meaningful as items move (Priority: P3)

An operator reorders and archives items freely over a document's life. They need every item's identifier to remain stable and never to imply a position — so cross-references survive a reshuffle and no identifier becomes misleading once items move out of sequence.

**Why this priority**: This is a correctness guardrail the operator called out explicitly. It is cross-cutting (part of well-formedness) rather than a standalone workflow, so it is verified last — but it is non-negotiable, not optional.

**Independent Test**: Author a document with an ordinal-looking identifier and confirm the engine rejects it loud; author valid identifiers, reorder and archive/unarchive items, and confirm every identifier is byte-for-byte unchanged throughout.

**Acceptance Scenarios**:

1. **Given** a document whose grammar would admit an item identified as `F3` (or a bare number, or `phase-2`), **When** the engine validates it, **Then** validation fails loud naming the ordinal-looking identifier as the cause.
2. **Given** a valid document, **When** items are reordered or archived and unarchived, **Then** every item's identifier is unchanged (identity is decoupled from position).
3. **Given** a document and its archive, **When** a new item reuses an identifier already present in either, **Then** validation fails loud on the duplicate.

### Edge Cases

- **No grammar declared**: a document with neither an embedded grammar block nor a resolvable frontmatter reference is not governable → fail loud, no writes (FR-001, FR-010).
- **Both embedded and referenced grammar present**: a single embedded grammar wins; the reference is ignored (FR-001 precedence).
- **More than one embedded `doc-grammar:` declaration**: which one is the declaration is ambiguous → **fail loud, no writes** (NOT "first wins") (FR-001, FR-010).
- **Parse failure**: the block stream does not parse against the declared grammar → fail loud with the offending span; archive/curate do nothing (FR-003).
- **Reconciliation hook declared**: `curate` validates its presence and reports "declared, not executed" — execution is out of scope for this feature (a separate feature owns it) (FR-008). It is neither run nor treated as drift in this feature.
- **Archive file missing on first archive**: created with frontmatter (FR-006); for the FR-005 uniqueness union, the ledger is empty so the union is just the live document.
- **Corrupt/unparsable archive file — two distinct cases** (an unreadable LEDGER is not the same as a corrupt archive BODY):
  - **Archive BODY unparsable/corrupt** (the archived Unit content): does not block live-document validation — archived identifiers for the uniqueness union come from the **ledger only** (FR-005), never from parsing or scanning the archive Unit contents; the body corruption is surfaced via the `curate` coherence check (FR-006), **not** a live-parse failure.
  - **Archive LEDGER unreadable/absent**: the ledger is the authoritative source for the uniqueness union (FR-005) and for locating Units to unarchive (FR-007), so its absence/unreadability has consequences a body corruption does not. On **`unarchive`** it is a **fail-loud locate failure** (FR-007/FR-010 — no entries to search, the engine cannot locate the requested Unit to restore). On **`curate`** it is surfaced as a **coherence NOTICE** (the ledger can't be verified against the archive contents), never fail-loud. On **`archive`** (first archive or append) an absent ledger is simply the empty-union state (the union is just the live document, per the "Archive file missing" edge) — only an *unreadable* (present-but-corrupt) ledger surfaces as a `curate` NOTICE.
- **Unarchive identity collision**: an item whose identifier already exists in the live document cannot be unarchived → fail loud (FR-007).
- **A live item cross-references a now-archived item**: the reference still resolves by identifier because identity is stable and the archived item keeps its identifier as its reserved-level structural marker in the archive file (FR-005, FR-006).
- **An item with no recognized status**: the grammar's status vocabulary does not match → parse/validation failure, fail loud (FR-003, FR-004).
- **A write fails partway through `--apply`** (archive, unarchive, or curate): the **durability promise** holds — content is never silently lost or corrupted. Because governed documents are version-controlled (Assumptions), any inconsistency the interruption leaves is **recoverable** (revert to the last commit + re-run) and **detectable** (the `curate` coherence check surfaces it as a NOTICE). The spec does not promise the operation is atomic; the write/recovery protocol is an implementation concern (FR-006, FR-010, Assumptions).
- **An operator hand-edits an archived Unit's identifier** (the archive is now stale *before* the next run): a subsequent `archive --apply` / `unarchive --apply` is **not blocked** by the pre-existing mismatch and does **not** auto-repair it — it guarantees coherence only for the Units that operation moves (`archive` appends; `unarchive` removes the named Unit from the archive — FR-006/FR-007); the stale entry is left in place. The `curate`-owned coherence check **surfaces** the ledger staleness as a NOTICE on the next `curate` run; the engine does not auto-repair or refuse the edit (FR-006; out of scope to actively guard).
- **Shallower-than-reserved heading after the first Unit**: a heading at a level shallower than (or peer-but-non-conforming to) the reserved Unit level appearing **after** the first Unit is neither preamble (the preamble region ends at the first Unit) nor a valid Unit body block (bodies are strictly deeper) → **fail-loud parse failure** (FR-003), naming the offending span and pointing the author to relocate the content into the preamble. A governable document is exactly *preamble THEN Unit sequence*; there is no well-formed interleaved-shallow-heading region (FR-002).
- **Empty / zero-Unit document**: a document that declares a valid grammar and valid frontmatter but parses to **zero Units** (a fresh roadmap with grammar + frontmatter but no rows; or a document whose Units have all been archived — the fresh-install state of the two proof documents before migration content lands) is **always vacuously well-formed** (FR-002/FR-003) — it is NOT a parse failure. A zero-Unit document has **no first Unit marker**, so under FR-002's two-region model **every block is preamble**; there is no "block that failed to fall into the preamble," so the all-preamble (zero-Unit-marker) case has **no fail-loud path** — it is vacuously well-formed, full stop. The fail-loud path (FR-003) is reserved for a **different** document: one that **does** have ≥1 Unit marker followed by a block that does not conform (e.g. a shallower-than-reserved heading after the first Unit) — the path FR-002's two-region model actually supports. The only, non-silent safety signal for the zero-Unit case is that the engine **reports the parsed Unit count** — `curate`/`archive` surface "N Units parsed" (here, "0 Units parsed") — so an operator who expected Units but got zero sees it; this is a reported count, **never a heuristic** that guesses intent (FR-003). `archive` and `curate` **no-op cleanly**: `archive` selects nothing (no archivable Units), `curate` is vacuously well-ordered and well-formed. `unarchive <id>` **fails loud** (FR-007/FR-010): a zero-Unit document has an empty (or absent) ledger, so the requested `--id` cannot be located — a fail-loud locate failure, not a silent not-found. No writes from any verb.

## Requirements *(mandatory)*

> Authoring note (applying this session's DRY-for-prose lesson): each rule is stated **once** as its canonical FR; later requirements and the Success Criteria **reference** the canonical FR rather than restating it.

### Functional Requirements

- **FR-001 (governable document — canonical)**: A document is **governable** only if it declares a resolvable block-level grammar, and a governable document has **exactly one** embedded grammar declaration. Resolution order, fail-loud with **no fallback**: (1) an **embedded** grammar block — the HTML comment whose first line/token is the `doc-grammar:` sentinel (e.g. `<!-- doc-grammar: peg … -->`), invisible in rendered markdown; this sentinel is **how the engine recognizes the grammar declaration among a document's possibly-many HTML comments** — any HTML comment not beginning with the `doc-grammar:` sentinel is an ordinary block, not a grammar declaration. If **more than one** HTML comment begins with the `doc-grammar:` sentinel, that is an **ambiguous declaration → fail loud** (FR-010) — it is **NOT** "first wins" / silent. Else (2) a **frontmatter reference** to a shared grammar by id, resolved against the project grammar-override location then the plugin's built-in defaults; else (3) **fail loud** ("document declares no grammar; not governable"). When **one** embedded block and a frontmatter reference are both present, the embedded block takes precedence. Resolved grammars are **trusted local config** run in-process (Assumptions — grammar trust model).
- **FR-002 (document model — canonical)**: The engine parses a governable document along one shared, type-agnostic pipeline: raw markdown → a **block stream** (via a standard markdown block parser) → a **pre-parse normalization step** that **excises engine-level document chrome** from the block stream — specifically (a) the embedded grammar-declaration comment — the single HTML comment beginning with the `doc-grammar:` sentinel (FR-001; more than one is the FR-001 fail-loud ambiguity, never silently excised), when present; other HTML comments are ordinary blocks and are NOT excised — and (b) the document frontmatter — so that chrome is never handed to the grammar → the document's **declared grammar**, operating over the remaining block stream and treating prose bodies as **opaque** → a typed tree of **Units**, each carrying `identifier`, `status`, `orderKey`, `span` (source line range), and an opaque `body`. The excised chrome is **not a Unit** and can never cause a parse failure; the grammar author never accounts for it. The engine recognizes the standard markdown block kinds — headings (ATX and Setext), paragraphs, lists (incl. nested items), tables, fenced/indented code, blockquotes, thematic breaks, and HTML blocks; each grammar decides which block kinds are **unit-structural** versus part of an opaque `body`. A block the grammar does not account for — after chrome excision *and* outside the document preamble (below) — is a parse failure (FR-003), not a silent skip.
  - **Two regions: preamble THEN the Unit sequence.** After chrome excision, a governable document has exactly **two regions, in order**: an optional **preamble** (everything **before the first Unit marker**) followed by the **Unit sequence** (from the first reserved-level Unit marker onward). The preamble is human-facing front matter — the document's title heading, intro prose, and any overview/Clarifications region, often at heading levels **shallower than the reserved Unit level**. The engine **preserves the preamble verbatim**, never treats it as a Unit, and **never fails loud on it** (so a document's own title `# Roadmap` above `### `-keyed Units does not trip FR-003). Within the **Unit sequence**, only two block shapes are valid: (a) reserved-level Unit markers and (b) their **strictly-deeper** Unit bodies. A heading that is **shallower than — or peer-but-non-conforming to — the reserved Unit level appearing AFTER the first Unit is a parse failure (FR-003)**, not a third region: such interleaved content does not fit the grammar, and the author must move it into the preamble or restructure the document. **Preamble**, **Unit**, and **Unit body** are the three exhaustive region kinds (after chrome excision), and they appear strictly as *preamble, then a sequence of Units-with-bodies* — there is no well-formed "shallow heading interleaved among Units" case. `archive`, `unarchive`, and `curate` operate **only on Units**: `curate` reorders **only Units** and leaves the **preamble preserved verbatim** (never reordered); `archive`/`unarchive` leave the preamble untouched.
  - **Unit boundary rule (where a Unit ends).** Because a Unit's `body` is opaque, every grammar MUST delimit Units by a structural marker that a Unit body provably **cannot** contain — otherwise the engine cannot tell where one Unit's body ends and the next Unit begins. Two concrete forms: (a) a **heading-keyed grammar** (e.g. the inbox, where a Unit IS a `### Title`) **reserves its Unit-heading level** — a Unit's opaque body may contain only **strictly deeper** heading levels (and non-heading blocks), and a heading at the reserved level **always starts a new Unit**; the Unit therefore spans from its reserved-level heading to the next reserved-level heading (or end of document). Headings **shallower** than the reserved level are **document preamble** (above) **only when they appear before the first Unit** — so a title `# Roadmap` above `### `-keyed Units is preamble, never a parse failure. A shallower-than-reserved heading that appears **after the first Unit** is **not** preamble (the preamble region has ended) and is **not** a valid Unit body block (bodies are strictly deeper); it is a **parse failure** (FR-003) — the author must relocate it to the preamble. (b) a **non-heading-keyed grammar** (e.g. the table-row-keyed roadmap) delimits Units by its own structural marker — the table row — which a body cannot contain. This boundary marker is what produces each Unit's `span`.
- **FR-003 (well-formed — canonical)**: A document is **well-formed** if and only if **both**: (a) its block stream **parses** against its declared grammar, **and** (b) the **FR-005 identifier invariants hold** (unique across document ∪ archive, human-readable, non-ordinal). Well-formedness is a determination over (a) parse success/failure plus (b) identifier validation, not a heuristic. Any failure of (a) or (b) **fails loud** and names the offending span (parse failure) or the violating identifier (invariant violation); no partial operation proceeds. (This is why FR-005 enforces its invariants "as part of FR-003 well-formedness": a document that parses cleanly but duplicates an archived identifier — or carries an ordinal-looking one — is **not** well-formed.)
- **FR-004 (status & order — canonical)**: A grammar declares its **status vocabulary**, which statuses are **terminal** (archivable), and an **order key**. A Unit is **archivable** if and only if its status is in the terminal set. The order key MUST be expressible over status and human-readable fields and MUST NOT be a **positional/sequence ordinal** (so reordering can never break identity). Ordering by a category/attribute that happens to also appear in a structured identifier (e.g. roadmap ordering by `phase`, part of `<phase>/<slug>`) is allowed. **Declared ordering relation (not lexicographic):** the order key names *which field sorts*, but for a **categorical/enum field** the value domain has no natural string order (roadmap `phase` = {design, plan, impl, multi} sorts design < plan < impl < multi, which is NOT alphabetical; `status` = {planned, in-flight, shipped, …} has no lexicographic meaning). Therefore each grammar MUST **declare an explicit ordering relation over the order-key field's value domain** — an **ordered enumeration** of that field's values (e.g. roadmap `phase` order = `[design, plan, impl, multi]`). A document is **well-ordered** if and only if its Unit sequence matches the **declared ordering relation**; **lexicographic comparison is NOT assumed** and is never the default. A grammar whose order key references a categorical field **without a declared ordering relation is a grammar error** (the relation is required whenever the order key uses a categorical field). **Tie-break (total ordering):** when two or more Units share the **same rank** under the declared ordering relation (equal order-key values), ties are broken **by identifier** — a stable, total secondary sort (identifiers are unique per FR-005, so this always yields a single definite order). The identifier tie-break applies **only within equal-rank values** and makes `curate` reorder and `unarchive` reinsertion position deterministic. It is a deterministic secondary sort, **not** a positional encoding — it does not reintroduce the non-positional-order-key prohibition above.
- **FR-005 (identifier invariants — canonical)**: A Unit's **identity is decoupled from its position**. The engine enforces these as universal invariants for every governable document, as part of FR-003 well-formedness, fail-loud on violation:
  - **Unique** within the document ∪ its archive (so unarchive — FR-007 — cannot collide). Uniqueness is a **case-sensitive exact match**. Archived identifiers for the union come **solely from the provenance ledger** (which is keyed by identifier — FR-006); the **archive contents are NOT a union input** (a Unit body may itself contain headings, so scanning the archive cannot distinguish a Unit-identifier marker from a body heading without the grammar parse the engine is forbidden to run on the archive — it would yield false collisions). The archive's **structural markers** (the reserved-level heading for a heading-keyed grammar, or the row for a row-keyed grammar) serve only as the FR-006 coherence cross-reference (marker vs ledger), never as a source of identifiers for this union. When the ledger is empty or its archive file is absent (first archive), the union is just the live document; a corrupt/unparsable archive does not block live-document validation — it is surfaced via the coherence check (FR-006), not by failing the live parse.
  - **Human-readable**: a single visible name *is* the identifier; there is **no parallel opaque token** (no UUID alongside it). The name's concrete shape is the grammar's choice (see "Per-grammar production" below) — a strict slug (`<phase>/<slug>`) or a readable title — never a cryptic token.
  - **Non-ordinal**: the engine **rejects** an identifier that **IS a positional/sequence index** — not merely any identifier that begins with a digit. The **v1 denylist is a closed set** of sequence-index shapes: `F<n>`; `phase-<n>`; `step-<n>`; `#<n>`; a **bare integer that is the entire identifier** (e.g. `2`); and a **leading enumeration marker** (`1.`, `3)`). It does **NOT** reject a prose title that merely happens to start with a number — `3 ways to industrialize execution` and `5 hard problems` are allowed, because the number is part of the prose, not a positional index of the Unit. "Refinable" means future versions may *add* patterns — the v1 contract is exactly this set, not an open-ended judgment.
  - **Per-grammar production (property-based, not a fixed shape)**: the engine universally enforces the *properties* (unique / non-ordinal / human-readable); each grammar declares its concrete identifier production — a strict slug (roadmap → `<phase>/<slug>`, the recommended shape) or a readable title (inbox). The engine does NOT mandate slug-shape; "slug" is a recommendation, the properties are the contract.
- **FR-006 (archive — canonical)**: The `archive` primitive parses a governable document (FR-002), selects the archivable Units (FR-004), **cuts** them by `span`, **appends** them to a sibling archive file (created with frontmatter if absent), and updates a **provenance ledger** (recording what moved and when, keyed by **identifier** — never by an ordinal range). The **ledger is the SOLE source** of archived identifiers for the FR-005 uniqueness union (it is keyed by identifier). The ledger lives **in the archive file**, not the live document — the live document carries **zero** archive bookkeeping (preserving the lean-live-document goal).
  - **Archive file structure — the archive file mirrors the live document's structural container per the grammar.** Archived Units are delimited in the archive file by the **same reserved-level structural marker as the live document** (FR-002 Unit boundary rule), so an archived Unit spans from its identifier marker to the next same-level marker (or end of file). This boundary is **unambiguous** precisely because a Unit body provably cannot contain that marker. The concrete container depends on the grammar's Unit-boundary form:
    - **Heading-keyed grammar**: archived Units are **reserved-level sections** (already specified above) — each archived Unit is a reserved-level heading and its body, spanning to the next reserved-level heading or end of file.
    - **Row-keyed grammar**: archived rows live in a **single markdown table that reproduces the live document's header row + separator row + column schema**; that one table IS the archived-Unit table, and each archived Unit is one row within it. An archived Unit's boundary is the row.
    The **provenance ledger occupies its own distinct section** (e.g. its own HTML-comment block or its own heading) **separate** from the Unit table/sections, so a scanner never confuses ledger rows with Unit rows (or a ledger heading with a Unit heading). The ledger entry (keyed by identifier) is therefore **sufficient to locate the Unit** for extraction or removal (FR-007); **no span/position field is stored** in the ledger — the boundary is structural, recovered by scanning the archive file for the matching identifier marker (the reserved-level heading, or the table row in the archived-Unit table) and its next same-level marker. **Durability promise (FR-010): an interrupted or failed `archive --apply` NEVER silently loses or corrupts content.** Because governed documents are **version-controlled** (Assumptions), any inconsistency an interruption leaves is fully **recoverable** — the operator reverts to the last commit and re-runs — and the **`curate` coherence check detects** such inconsistency and surfaces it as a NOTICE. The spec states only this promise; the concrete write ordering, temp/rename, and crash-handling **protocol is an implementation concern** pinned by the contracts + RED tests, not specified here. `archive --apply` writes **append-only** to the archive (it appends the moved Units and their ledger entries; it never rewrites prior archived Units). `unarchive --apply` is **NOT append-only** to the archive: it **removes** the named Unit (and its ledger entry) from the archive file and reinserts the Unit into the live document (FR-007). Both **guarantee coherence for the Units that operation moves**: the operation's own moves leave the ledger matching the archive file's actual contents for those Units. A **pre-existing mismatch** (e.g. from a prior manual identifier edit that left the ledger stale before this run) is **neither auto-repaired nor blocking** — the run leaves it in place and does not fail. The **coherence check** that surfaces such drift is owned by the **`curate` verb** (the health-check verb, FR-008): it cross-references each ledger identifier against the **structural markers** present in the archive file (the reserved-level heading for a heading-keyed grammar, or the row for a row-keyed grammar — NOT a generic heading scan, and NOT a uniqueness-union input — FR-005) and reports any mismatch (e.g. ledger/marker mismatch from a manual identifier edit) as a **`curate` report line — a NOTICE, not a fail-loud failure**. A **corrupt archive BODY** (unparsable archived-Unit content) likewise does **not** block live-document validation and is surfaced only via this `curate` NOTICE. An **unreadable or absent LEDGER** is treated differently because the ledger is authoritative for both the uniqueness union (FR-005) and Unit location (FR-007): on `curate` an unreadable ledger is surfaced as a coherence **NOTICE** (the ledger can't be verified), while on `unarchive` an unreadable/absent/empty ledger is a **fail-loud locate failure** (FR-007/FR-010 — no entries to locate the requested Unit). Manually editing a Unit's identifier marker after archiving is the **operator's responsibility**; the coherence notice **surfaces** the resulting ledger-vs-marker staleness, but the engine does not refuse or auto-repair manual edits — and a pre-existing mismatch does not block a subsequent `archive`/`unarchive` run.
- **FR-007 (unarchive)**: The `unarchive` primitive is the symmetric reversal of FR-006: it **locates** the named Unit via its ledger entry (keyed by identifier — FR-006) and **lifts** it from the archive file by scanning for the matching identifier marker and reading to the next same-level structural marker (the FR-006 archive file structure makes this span unambiguous — no stored position is needed), returns it to the live document — reinserted at its **declared-order position** — and removes its ledger entry. **"Declared-order position" is relative to the live document's CURRENT neighbors**: the engine inserts the Unit where the grammar's declared ordering relation (FR-004 — order key + tie-break by identifier) places it **among the Units actually present in the live document**, wherever those Units currently sit. It is **NOT** computed against a hypothetical fully-sorted document, and **`unarchive` does NOT reorder the rest** — it touches only the one Unit it reinserts (curate is the verb that imposes order on the whole document; FR-008). It **fails loud** (FR-010) on a **locate failure** — the provenance ledger is **absent, empty, or unreadable**, or contains **no entry for the requested `--id`** (the engine cannot locate the Unit to restore) — and on an identity collision against the live document (FR-005 uniqueness). A missing/empty/unreadable ledger is a fail-loud locate failure distinct from the `--id`-not-in-archive case only in that the ledger itself provides no entries to search. (A corrupt archive **body** — as opposed to an unreadable ledger — does NOT make `unarchive` fail loud when the ledger still locates the Unit; it is a `curate` coherence NOTICE — FR-006.) An archive→unarchive round-trip is **content-equivalent** (the Unit returns with body and identity intact, placed at its declared-order position relative to current neighbors) — **not byte-identical** (intervening cuts or edits make byte-restoration neither possible nor required) — and **well-ordered IFF the live document was already well-ordered before the round-trip** (a single-Unit insertion preserves an already-well-ordered sequence but does not impose order on an already-unordered one; FR-013 permits a migrated document to be unordered). The ledger does not record position; reinsertion order is derived from the grammar relative to current neighbors.
- **FR-008 (curate — canonical)**: The `curate` primitive is a **composable primitive whose invocation context is intentionally undecided**. It ensures three properties: **well-formed** (FR-003); **well-ordered** (FR-004 — and on `--apply`, reorders mechanically without changing any identity per FR-005); **properly archived** (composes `archive --apply`, FR-006 — on `--apply`, archivable Units are moved to the archive, not merely reported). `curate` also **owns the FR-006 coherence check** (ledger vs archive-file identifier markers), reporting any mismatch as a **NOTICE report line, never a fail-loud failure** (manual identifier edits are the operator's responsibility, not refused — FR-006). On `--apply` curate **reorders first, then archives** (writing the live document for the reorder, then composing `archive --apply`). It makes the same **durability promise** (FR-010): an interrupted or failed `curate --apply` NEVER silently loses or corrupts content — any inconsistency it leaves is **recoverable** via version control (revert + re-run) and **detectable** via the coherence check (FR-006, Assumptions). It does **not** promise the whole operation is atomic. A fourth property, **up-to-date**, is a **declared seam only** in this feature: a grammar MAY declare a reconciliation hook and `curate` recognizes/validates its presence, but **reconciliation execution is out of scope here** — the up-to-date check is a no-op-with-notice (reports "declared, not executed" when present; silent when absent). A subsequent feature implements execution against the seam.
- **FR-009 (dry-run default)**: `archive`, `unarchive`, and `curate` default to **dry-run** (report planned changes, write nothing); writing requires an explicit `--apply`.
- **FR-010 (fail-loud / no fallbacks — canonical)**: Every **validation failure** fails loud with an actionable message and performs **zero writes**: an ungovernable document (FR-001), an **ambiguous grammar declaration** (more than one `doc-grammar:` embedded comment — FR-001), a parse failure (FR-003), an identifier-invariant violation (FR-005), and an unarchive collision (FR-007). For these validation-failure modes **"zero writes" is absolute** and spans **both files** (the live document and the sibling archive file): no empty/partial archive file is created and no partial document mutation is left behind. For the **`--apply` write path**, the guarantee is the **durability promise** (not absolute atomicity): an interrupted or failed `archive`/`unarchive`/`curate --apply` **NEVER silently loses or corrupts content** — because governed documents are version-controlled (Assumptions), any inconsistency an interruption leaves is **recoverable** (revert to the last commit + re-run) and the `curate` coherence check **detects** it and surfaces it as a NOTICE (FR-006). The concrete write/ordering/recovery protocol is an implementation concern pinned by the contracts + RED tests, not specified here. An "actionable message" names the specific absent/offending element (the unresolved grammar, the offending span, the violating identifier). No silent skips (other than the explicitly-deferred up-to-date check in FR-008) and no mock/placeholder data.
- **FR-011 (anti-coupling invariant)**: The **shipped product mechanism** contains **zero references to the predecessor lifecycle plugin** — no import, no shell-out, no "ported-from"/"mirrors" prose. The archive mechanism is reimplemented fresh and described entirely on its own terms. This invariant is **machine-checked** and release-blocking (non-zero exit fails the gate):
  - **Scan scope** — the product mechanism only (engine code, command verbs, skill bodies, grammars): `plugins/stack-control/src/document-model/**`, the three verb modules, `plugins/stack-control/skills/**` (covers all three skills — `archive`, `unarchive`, `curate`), `plugins/stack-control/grammars/**`, and the feature's test fixtures.
  - **Match pattern** — a **closed set** (like FR-005's denylist): the single literal token **`dw-lifecycle`** — which is simultaneously the predecessor plugin name, its CLI binary name, and its skill-namespace prefix (`dw-lifecycle:`). Matched as a **whole token** (word-boundary), **case-insensitive**. The word-boundary constraint means it does NOT match the substring "lifecycle" inside `pluggable-lifecycle-providers`, `stack-control` prose, or any other use of "lifecycle" — only the bare `dw-lifecycle` token.
  - **Exclusions** — `specs/**`, design docs, provenance notes, **and the two proof documents (`ROADMAP.md`, `DESIGN-INBOX.md`)** are NOT scanned; they are governed *content* that legitimately names the predecessor as lineage (the plugin-local roadmap is migrated succession content — its program framing is "absorb-then-retire" the predecessor), exactly as `specs/**` and design docs do (Assumptions).
- **FR-012 (grammar distribution)**: Shared grammars ship as **built-in defaults** within the plugin and are **project-overridable** at a project grammar location, using the project's established override-resolution pattern. The per-document resolution order in FR-001 selects among embedded → project override → built-in default. A project override is **trusted local config** run in-process — the same trust model as `.deskwork/*.ts` overrides (Assumptions — grammar trust model).
- **FR-013 (two proof documents)**: The feature establishes **two structurally different** governable documents as its real instances and dogfood, each with a **fully enumerated status vocabulary** and a **declared ordering relation over its order-key domain** (FR-004 — never lexicographic):
  - (a) a **title-keyed design inbox** — statuses `captured` (active), `promoted` (terminal), `dropped` (terminal). Its **declared ordering relation** orders by status rank `[captured, promoted, dropped]`, with the within-rank tie-break by identifier (the title) per FR-004 — so active (`captured`) entries sort ahead of terminal ones and equal-status entries are ordered by title.
  - (b) a **`<phase>/<slug>`-keyed roadmap** — statuses `planned`, `in-flight` (active), `shipped`, `cancelled`, `retired` (terminal). Its order key is `phase`, with the **declared ordering relation** `phase` order = `[design, plan, impl, multi]` (explicitly NOT alphabetical); equal-`phase` Units tie-break by identifier per FR-004. The roadmap grammar **declares a reconciliation hook** — the seam the future roadmap-discipline protocol (Out of Scope) plugs into — so `curate` recognizes/validates it and reports it as "declared, not executed" (FR-008 seam-only scope: the hook is recognized but never executed in this feature). This gives US2's Independent Test and Acceptance Scenario 4 a committed fixture: a built-in grammar that actually declares a hook.
  Both are governed by the **same engine**, differing only in their grammars (SC-005). Establishing these documents means giving each structure, a grammar, and migrated content. **Migration contract**: "**lossless**" is defined over **content bodies** — every existing entry/row appears in the new governed document and no body content is dropped. It is **not** defined over identifier spelling: when a pre-existing nonconforming identifier (e.g. a leading number, `#<n>`) must be **normalized** to satisfy FR-005, migration may do so. Migration is **manual authoring by the implementer** — a **one-time establishment step, not a shipped tool and not ongoing bookkeeping**: the implementer **records each such rename by hand in the establishment commit/PR message** so the provenance isn't lost — it is **not** a tool output, **not** persisted in the live document (which carries zero bookkeeping), and **not** persisted in the archive ledger (no ledger schema change). There is no `migrate` module and no migration scan. The migration bar is **well-formed** — the migrated document MUST **parse against its grammar with valid identifiers** (FR-003). It need **not** be fully-curated: it MAY contain settled (terminal-status) items that are not archived (a subsequent `curate`/`archive` handles those) and MAY be unordered. Well-formed (the migration bar) is distinct from fully-curated (a separate step that also reorders and archives); requiring the migration to "pass `curate`" would wrongly demand it archive settled items, so the bar is well-formedness only. It does **not** include the protocols that govern when their contents change (see Out of Scope).

### Key Entities

- **Governable Document**: a markdown document that declares a resolvable grammar (FR-001); the unit of operation for archive and curate. After chrome excision it has exactly two regions in order — an optional **preamble** (everything before the first Unit marker: title/intro + shallower-than-reserved-level headings, preserved verbatim, never a Unit) followed by the **Unit sequence** (Units and their strictly-deeper Unit bodies). A shallower-than-reserved heading appearing *after* the first Unit is a fail-loud parse failure, not preamble (FR-002, FR-003).
- **Grammar**: the declared, block-level description of a document's structure — its Unit production, status vocabulary, terminal set, order key, identifier production, and optional reconciliation hook. Lives embedded in the document or as a shared, overridable file (FR-001, FR-012).
- **Unit**: a parsed item of a document — `identifier`, `status`, `orderKey`, `span`, opaque `body` (FR-002). The thing that is ordered, archived, and referenced.
- **Archive File**: the sibling document that receives archived Units, delimiting each by the **same reserved-level structural marker** as the live document (FR-002 boundary rule) so a Unit's span is unambiguous on extraction (FR-006, FR-007).
- **Provenance Ledger**: the record (held **in the archive file**, not the live document) of what was archived and when, keyed by identifier; it is the **sole source** of archived identifiers for the FR-005 uniqueness union and is sufficient to locate a Unit for unarchive; kept coherent with the archive file's contents — the cross-reference is against the archive's **structural markers** (reserved-level heading for heading-keyed grammars, the row for row-keyed grammars), used for the coherence check only, never as a union input (FR-006).
- **Reconciliation Hook**: an optional grammar-declared seam for an external source of truth. In this feature `curate` only *recognizes/validates* its presence; *executing* it (running the source, computing drift) is out of scope for this feature — a separate feature owns it (FR-008).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An operator can move every settled item out of a live document into its archive in a single command; afterward the live document contains **zero** archivable Units (verifies FR-006 / US1).
- **SC-002**: After `curate --apply`, the document **parses** against its grammar, its Units are in the **declared order**, and any archivable Units have been **moved to the archive** (`--apply` reorders then archives — FR-008) (verifies FR-003 + FR-004 + FR-006 / US2).
- **SC-003**: Two distinct guarantees. **(i) Validation failures** (ungovernable document, parse failure, invariant-violating identifier, unarchive collision) produce a loud error and **zero writes** — no empty/partial archive file, no partial document mutation (verifies FR-010 fail-loud). **(ii) An interrupted `--apply`** leaves **no silently-lost content**, and any inconsistency it leaves is **detectable** and **recoverable**: after a simulated interruption, the `curate` coherence check reports the ledger/archive mismatch as a NOTICE and a version-control revert restores the pre-operation state (verifies FR-006 + FR-010 durability promise). This does **not** assert "no partial mutation" as an absolute.
- **SC-004**: An ordinal-looking identifier is **rejected**, and every identifier is **unchanged** across any reorder and any archive→unarchive round-trip (verifies FR-005 / US3).
- **SC-005**: The two proof documents — a title-keyed inbox and a `<phase>/<slug>` roadmap — are governed by the **same engine** with **only their grammars differing** (verifies FR-013 generality).
- **SC-006**: A scan of the shipped product **mechanism** (engine code, command verbs, skill bodies, grammars — NOT governed content like the two proof documents) returns **zero** predecessor-plugin references (verifies FR-011).
- **SC-007**: An archive→unarchive round-trip restores a **content-equivalent** state (the Unit returns with body and identity intact, reinserted at its declared-order position **relative to the live document's current neighbors** — not byte-identical), **and well-ordered IFF the live document was already well-ordered before the round-trip** (a single-Unit reinsertion preserves an existing order discipline but does not impose order on an already-unordered document — `curate` does that); and after every archive run the provenance ledger matches the archive file's contents **for the Units that run moved** (a pre-existing manual-edit mismatch is left in place, not repaired or blocking) — the invariant `curate` checks and reports on (as a NOTICE) per FR-006/FR-008 (verifies FR-006 coherence + FR-007 reversibility).

## Assumptions

- **Users are operators and agents**; this is developer tooling that ships in the stack-control plugin (command verbs + thin skills), not project-specific scaffolding. It is dogfooded against the project's own documents.
- **Markdown is the document format**, and a standard markdown block parser produces the block stream the grammar runs over (FR-002).
- **The concrete grammar/parser technology is deferred** to the planning phase's research step; this spec commits only to "a formal grammar compiled to a real parser, run over the markdown block stream."
- **The predecessor lifecycle plugin's workplan-archive is the conceptual origin** of the archive mechanism but is neither imported nor referenced by the product (FR-011). This is the only place that lineage is named.
- **The two proof documents are established by this feature** (structure + grammar + migrated content); migration is manual authoring by the implementer (a one-time establishment step, not a shipped tool) and is lossless over **content bodies** (identifiers may be normalized to satisfy FR-005, with each rename recorded by hand in the establishment commit/PR message — not a tool output, not persisted in the live document or ledger; the result must be **well-formed** — parses against its grammar with valid identifiers, FR-003 — but need not be fully-curated: settled items may be archived by a subsequent `curate`/`archive`, and order is not required) per FR-013. The *protocols* that govern when their contents change are separate features (Out of Scope).
- **Strict typing and a per-file size ceiling** apply per the project constitution (Principle VI); tests are written test-first against real fixture document trees on disk (Principle I, II).
- **Governed documents are version-controlled, and version control is the durability/recovery backstop.** This is what makes the durability promise (FR-006/FR-008/FR-010) honest: an interrupted or failed `--apply` never silently loses content because any inconsistency it leaves is recoverable — the operator reverts to the last committed state and re-runs (version control is the recovery journal) — and the `curate` coherence check detects any residual mismatch and surfaces it as a NOTICE (FR-006, FR-010, SC-003). The concrete write/ordering/recovery protocol is deferred to implementation (plan.md + contracts + RED tests).
- **Grammar files are trusted local config.** A grammar (embedded or a `.stack-control/grammars/*.peg` override) is operator-authored project configuration compiled and run **in-process** when `archive`/`curate` run; because it can carry semantic actions, running it executes code from the repository. This is the **same trust model as the project's `.deskwork/*.ts` template/doctor overrides** — accepted for this local, single-operator tooling context. The engine does **not** restrict grammars to an action-free subset; sandboxing grammar execution is out of scope (referenced from FR-001/FR-012).

## Out of Scope

Named here so the boundary is explicit; each is a candidate subsequent feature.

- The **roadmap-discipline protocol** — when rows advance between statuses, and reconciliation against on-disk feature state. (Plugs into curate's FR-008 reconciliation hook.)
- **Reconciliation hook *execution*** — running a declared reconciliation source and computing drift. This feature declares and validates the seam (FR-008) but does not execute it.
- **Roadmap-canonizing skills** beyond the generic `archive`/`curate` primitives.
- The **concrete parser-generator choice** (deferred to the planning research step, per Assumptions).
- **Retiring or migrating** the predecessor plugin's workplan-archive. Generality is *proven* (FR-013) without touching the predecessor.
- **`curate`'s invocation wiring** (session-start, a future roadmap skill, etc.) — the primitive is deliberately unopinionated about when it runs (FR-008).
- **Active detection/refusal of manual identifier edits** — the coherence check surfaces resulting staleness (FR-006), but guarding against an operator hand-editing an identifier is not built.
- **Concurrent invocation** of two primitives on the same document — out of scope for this local, single-operator tooling; not guarded against.
