# Feature Specification: Govern the spec, not just the implementation

**Codename**: `design/spec-governance`

**Feature Branch**: `feature/pluggable-lifecycle-providers` (built on the shared program branch; not a per-feature branch)

**Created**: 2026-06-06

**Status**: Draft

**Input**: User description: "Govern the spec, not just the implementation — a stack-control capability that fires the cross-model audit-barrage over a SPEC at definition time, mirroring the existing deskwork-governance after_implement extension that fires the barrage over produced code."

## Why this feature exists (thesis grounding)

stack-control's organizing thesis is **invest heavily in up-front design and tooling; industrialize execution**. The design phase is the higher-leverage place to catch defects — a defect caught in the spec is orders of magnitude cheaper than the same defect caught after implementation. stack-control already fires a cross-model audit-barrage automatically *after implementation* (the founding `impl/governance` extension, `deskwork-governance`'s `after_implement` hook). This feature is the design-phase analog: extend governance **left**, to the spec itself.

The motivating evidence is concrete: a manual cross-model barrage run over the `impl/execution-engine` spec (`specs/002`, claude + codex) surfaced 51 findings — including **3 real contradictions the author had introduced** plus deep design gaps a single authoring pass missed. Spec quality must not depend on a human remembering to run the barrage. This is **detection over instruction**: make the failure state (an ungoverned, self-contradictory spec graduating to planning) mechanically surfaced rather than relying on discipline.

- Q: FR-012 — how is spec-governance delivered? → A: **A Spec Kit governance extension with hooks**, mirroring the founding `deskwork-governance` `after_implement` extension, so it fires **universally** (front-door skills *or* raw `/speckit-*`). NOT folded into the front-door skills only.

### Session 2026-06-07 (governance findings — self-hosted dogfood)

The feature governed its own spec (T024). Nine findings were lifted; their resolutions are encoded below.

- Q (AUDIT-01/-02, HIGH cross-model): the gate criterion is written in "HIGH/MEDIUM" but the spec overloads those across two orthogonal axes — **confidence** (cross-model agreement) and **severity**. Which does the gate count? → A: **Severity.** The ported convergence criterion counts finding **severity** (`blocking`/`high`/`medium`); **confidence** is a separate annotation renamed to `cross-model-agreed | single-model` so the two never collide. This dissolves AUDIT-02: a single-model HIGH-**severity** finding still blocks the gate (it does not become un-blockable just because only one family flagged it).
- Q (AUDIT-08): FR-002 says the barrage MUST run multiple families — is a single-family *outcome* valid? → A: **The barrage MUST attempt all configured families in parallel; a run with ≥1 COVERING family is a valid, successful audit recorded with honest reduced coverage (FR-008).** The floor is one covering family — liftable AND exit-0 (AUDIT-20260607-42); the cross-model-agreement signal is simply unavailable below two. Matches the as-built barrage + the 2026-06-01 "1 healthy model IS a successful audit" directive (with the post-AUDIT-20260607-42 refinement that a single non-zero-exit family does NOT clear the floor).
- Q (AUDIT-05): when both `after_clarify` and `after_plan` are enabled, how do the loop + ceiling compose? → A: **Independent per-checkpoint loops with independent ceilings.** Each checkpoint runs its own convergence loop over its own artifact set; passing the `after_clarify` gate is **durable** and is NOT re-opened by `after_plan` findings. The iteration ceiling (FR-014) is **per-checkpoint**, not global.
- Q (AUDIT-03): the two-consecutive-iteration branch requires only 0 HIGH — what happens to open MEDIUM findings at graduation? → A: **There are none open at graduation.** Per the convergence rule (FR-010), the slush step (FR-015) runs automatically before the gate and bins **all** still-open MED/LOW across the checkpoint's runs the moment the dampener engages, so no spec graduates carrying **open** MEDIUM findings. Slushed = not fixed, not open; preserved + burn-down-able via a manual, out-of-loop pass (FR-015); HIGHs are never slushed. (Corrects an earlier draft that said MEDIUMs are "carried open" — the faithful dw-lifecycle protocol slushes them.)
- Q (AUDIT-04): the barrage is non-deterministic — what is an "iteration" and what makes two "consecutive"? → A: **An iteration is one recorded barrage run** (one audit-log lift section). "Consecutive" = the last N recorded runs **for that checkpoint**, regardless of whether the spec text changed between them; an inter-iteration edit does NOT reset the count. Two-consecutive-quiet is a **stability heuristic**, not a determinism proof. The FR-014 ceiling counts recorded runs.
- Q (AUDIT-07): all configured families are available but **all** error/time out mid-run (zero covering) — clean run or failure? → A: **Failure.** A zero-COVERING run is treated identically to no-capability: **fail loud (FR-005), spec NOT recorded as governed** — distinct from a clean zero-*finding* run (FR-009, where ≥1 family ran to completion and found nothing). Per AUDIT-20260607-42 this explicitly includes the case where families emitted bytes but all crashed (non-zero exit) — bytes-without-completion is not coverage.
- Q (AUDIT-09): the Dependencies "and/or front-door skills" wording reopens the front-door-only path FR-012 forbids. → A: **The hook mechanism is mandatory;** front-door skills are listed only as callers that benefit from the universal hook, never as an alternative delivery surface.
- Q (AUDIT-06): SC-005's "one governance surface" vs the dw-lifecycle isolation constraint. → A: **Resolved by `multi/migrate-audit-barrage` (now done):** the barrage + protocol are vendored into stack-control, so spec-phase and implementation-phase governance share a single in-stack-control store — SC-005's single-surface claim is now literally true, with **no dw-lifecycle dependency** (FR-006 updated).

### Session 2026-06-07 round 2 (re-governance findings — definitions + integrity)

The first clarify round drove HIGH findings to zero; the re-barrage surfaced 8 MEDIUM/LOW completeness gaps (AUDIT-16..23), mostly under-specified versions of as-built behavior. Resolutions:

- Q (AUDIT-16): the Checkpoint entity lists `after_specify`, but no FR defines its artifact set. → A: **`after_specify` is out of scope for this feature** — only `after_clarify` (spec) and `after_plan` (spec+plan) are wired checkpoints (FR-011/FR-013). The entity enumeration is narrowed accordingly.
- Q (AUDIT-17): "healthy family" is load-bearing (fail-loud vs clean-run, coverage) but undefined. → A: a model family **COVERS** a run when it ran to completion and emitted output — **≥1 byte of stdout, no spawn/timeout error, AND exited 0**. Coverage (FR-008) and the zero-coverage outage (FR-005) count COVERING families. *(Refined by AUDIT-20260607-42, below: "healthy" is split into liftability — bytes>0 + no spawn failure, governing what the lift extracts — and coverage — liftability + exit-0, governing the FR-008 count and the FR-005 outage. The exit-0 clause closes the original AUDIT-17 gap where a crash-after-emitting-bytes family was counted as a covering "clean contribution," letting an outage masquerade as a clean run in the single-family floor.)*
- Q (AUDIT-20260607-42, HIGH — fail-loud integrity): the as-built `isModelRunHealthy` predicate (`stdoutBytes>0 && spawnError===undefined`) did NOT exclude a non-zero process exit. For the LLM CLIs the barrage drives (claude/codex/gemini headless), a non-zero exit usually means a failure (rate-limit, auth expiry, mid-stream drop); a family that prints a banner then dies `exit 1` satisfied the predicate and was counted a healthy family contributing a "clean" 0 findings — INDISTINGUISHABLE from a legitimate clean run in the single-family floor, defeating FR-005/US3/SC-003. → A: **Split the one conflated predicate into two.** **Liftability** (`stdoutBytes>0 && no spawn failure`, UNCHANGED) governs ONLY what the lift extracts — liftable output is NEVER discarded, so a non-zero-exit family that emitted real findings is still lifted when the run has coverage. **Coverage** (NEW, stricter: liftability **AND** `exitCode===0`) governs the FR-008 healthy-coverage count, the FR-005 zero-coverage OUTAGE determination, the clean-run claim, the summary line, and the tip.sha gate. `exitCode===0` also excludes timeout (`-1`) and spawn failure (`-2`), closing the latent gap where FR-008's prose said "no spawn/timeout error" but the code checked only `spawnError`. Net effect: a crash-after-banner family's bytes are still lifted in a mixed run, but it can no longer make a run governed-clean; only when EVERY family is non-covering is the run an OUTAGE (fail loud, no auto-lift; run-dir `.md` artifacts remain on disk for manual triage).
- Q (AUDIT-18): "same root cause" (the agreement basis) has no matching rule. → A: two findings cluster as the same root cause when their **headings share a ≥12-character substring** (case-insensitive, punctuation-stripped) **OR** they cite a **shared repo-relative path token**; clustering is transitive. Cross-model agreement = a cluster spanning ≥2 families (FR-003).
- Q (AUDIT-19): the FR-010/SC-007 "override" has no surface/authorization/format. → A: an override is an **operator action** recorded with a **mandatory reason** — supplied via `GOVERN_OVERRIDE` (env) / `--override "<reason>"` (gate flag) and recorded in the run verdict (`override.recorded` + `reason`). No reason → no override.
- Q (AUDIT-20): the `acknowledged` disposition can clear a blocking finding from the gate, bypassing "recorded override". → A: the gate counts only **open** findings, so clearing a HIGH-severity finding requires either a **fix** (recorded `fixed-<sha>`) or an **acknowledgment with a substantive recorded reason** — the same evidentiary bar as an override, never silent. The finding state machine enforces the recorded reason.
- Q (AUDIT-21): the per-checkpoint ceiling (FR-014) names no default/config surface. → A: default **5**; configurable per checkpoint via `--ceiling` / `GOVERN_CEILING`.
- Q (AUDIT-22): SC-006 is stated deterministically but the barrage is stochastic. → A: SC-006 is **probabilistic** — a seeded contradiction is surfaced with high probability across the model battery (the cross-model design raises detection odds); it is not a per-run determinism guarantee.
- Q (AUDIT-23): the unattended loop implies automated spec-fixing without naming the actor. → A: spec-governance **governs** (detects + gates); it does **NOT** auto-edit the spec. The "fix" step of the loop is the **author's** act (the agent/operator driving the define/clarify cycle). "Unattended" (FR-014) means the **gate** evaluates and bounds the loop without an operator present (recording non-convergence at the ceiling) — not that the spec is machine-edited.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Spec is automatically governed at definition time (Priority: P1) 🎯 MVP

A maintainer authors or revises a spec through the stack-control front door. When the spec reaches a defined definition-time checkpoint, the cross-model audit-barrage fires **automatically** — multiple model families review the spec in parallel — and surfaces findings, with no manual barrage invocation required. The maintainer sees the findings without having had to remember to ask for them.

**Why this priority**: This is the core capability and the whole point of the feature — automatic, mechanical spec governance. Without it, spec governance remains a discipline a human can forget. It delivers value on its own: even with the simplest possible finding presentation and no blocking gate, automatic surfacing is strictly better than the status quo (manual-and-forgettable).

**Independent Test**: Author a spec containing a deliberate self-contradiction, drive it to the defined checkpoint, and observe that a barrage run is recorded and the contradiction is surfaced as a finding — without any manual barrage command being issued.

**Acceptance Scenarios**:

1. **Given** a spec at the defined definition-time checkpoint, **When** the checkpoint is reached, **Then** a cross-model barrage run is initiated automatically and its findings are surfaced to the maintainer.
2. **Given** a spec with no detectable issues, **When** the barrage runs, **Then** a run is still recorded (an empty/clean result is a recorded outcome, not a skipped step).
3. **Given** the barrage has run, **When** the maintainer inspects the result, **Then** each finding is attributed to the model family that produced it.

---

### User Story 2 - Cross-model agreement is flagged as the high-confidence signal, findings routed to triage (Priority: P2)

When two or more independent model families flag the same root cause, that agreement is the strongest signal that the finding is real. The maintainer can see which findings have cross-model agreement (HIGH confidence) versus single-model findings, and route all findings into a triage workflow where each gets an explicit disposition.

**Why this priority**: Genetic diversity in failure modes is the reason to run *multiple* models rather than one; surfacing the agreement signal is what turns a pile of findings into a prioritized worklist. Builds directly on US1.

**Independent Test**: Run the barrage over a spec with a contradiction obvious enough that two model families both flag it; confirm the finding is labeled HIGH-confidence (cross-model agreement) and is distinguishable from single-model findings, and that each finding can be given a disposition.

**Acceptance Scenarios**:

1. **Given** a barrage result where ≥2 model families flagged the same root cause, **When** findings are presented, **Then** that finding is labeled HIGH-confidence / cross-model agreement.
2. **Given** a set of findings, **When** the maintainer triages them, **Then** each finding receives an explicit disposition that is recorded durably.
3. **Given** prior runs with dispositioned findings, **When** the barrage re-runs on a later revision, **Then** the prior runs' dispositions remain intact in their own lift sections and the new run is recorded as its own all-`open` section (no disposition is lost, and entries are not correlated across runs).

---

### User Story 3 - Governance fails loud when the audit capability is absent (Priority: P3)

Spec governance composes the existing audit-barrage capability in-house. If that capability is unavailable (no model families installed/reachable, or the barrage primitives are missing), the define flow MUST fail loud with an actionable message — it MUST NOT silently skip governance and let an ungoverned spec proceed as if governed.

**Why this priority**: A silently-skipped governance step is worse than no feature at all — it gives false assurance. This is the no-fallbacks principle applied to governance. Lower priority only because it is a guardrail on US1/US2, not the primary value.

**Independent Test**: Simulate an environment with no available audit capability, drive a spec to the checkpoint, and confirm the flow fails loud with an actionable message rather than reporting success or silently continuing.

**Acceptance Scenarios**:

1. **Given** no audit capability is available, **When** the checkpoint is reached, **Then** the flow fails loud with an actionable message and does NOT report the spec as governed.
2. **Given** some but not all model families are available, **When** the barrage runs, **Then** it proceeds with the available families AND records reduced coverage (degraded, but honest — never silently presented as full coverage).
3. **Given** every model family emits a banner then exits non-zero (rate-limit / auth-expiry / mid-stream drop — zero COVERING families despite bytes on disk), **When** the checkpoint is reached, **Then** the flow treats it as an OUTAGE and fails loud (AUDIT-20260607-42) — the emitted bytes do NOT make the run governed-clean, and the spec is NOT recorded as governed. (This is the crash-after-banner case the liftability/coverage split exists to catch.)

---

### Edge Cases

- **No model families available**: fail loud (US3) — never silent skip.
- **One model family available/covering**: a **valid, successful run** recorded with reduced coverage (FR-002/FR-008) — the cross-model-agreement *signal* is unavailable below two families, but single-model findings still surface and the gate still counts their **severity** (a single-model HIGH-severity finding still blocks). One COVERING family (liftable AND exit-0, AUDIT-20260607-42) is the floor.
- **A model family times out or errors mid-run (but ≥1 covers)**: the run records partial coverage with the failure noted; it does not abort the whole barrage as long as at least one family COVERED (exit-0). The timed-out / non-zero-exit family's emitted bytes are still LIFTED for findings (liftability) — they are just not counted toward coverage.
- **Crash-after-banner family** (AUDIT-20260607-42): a family that prints a banner then dies `exit 1` (rate-limit, auth expiry, mid-stream drop) is **liftable but NOT covering**. In a mixed run its bytes are lifted alongside the covering families; in the single-family floor it does NOT clear the floor — the run is a zero-coverage OUTAGE. This is the case that, before the liftability/coverage split, let an outage masquerade as a clean run.
- **ALL available families fail at runtime (zero covering)**: treated identically to no-capability — **fail loud (FR-005)**, the spec is NOT recorded as governed. This covers every family being a spawn error, a timeout, a non-zero exit, OR emitting zero bytes (AUDIT-20260607-42). It is distinct from a clean zero-*finding* run (FR-009): "zero findings because ≥1 family RAN TO COMPLETION (exit-0) and found nothing" is a recorded governed run; "zero findings because nothing covered" — including a family that emitted bytes then crashed — is an outage, never a clean result.
- **Spec re-revised after triage**: prior runs' dispositions are preserved in their own lift sections; the new run produces a fresh all-`open` lift section (it does NOT correlate entries across runs — a still-present defect re-appears as a new `open` entry); a re-run is never pre-emptively skipped as a no-op (empty revisions beat missed changes).
- **Barrage latency vs. the unattended directive**: the barrage can be slow and the gate is iterative (FR-010/FR-014). "Unattended" means the **gate** evaluates, slushes, and bounds the loop without an operator present — NOT that the spec is machine-edited (FR-014 no-auto-edit). The remediation (the "fix" between barrages) is the authoring agent/operator's act under the normal spec-editing contract; the loop remains bounded (FR-014) so a non-converging spec escalates rather than runs forever.
- **Governance never converges**: findings keep surfacing across iterations — the loop hits the configured iteration ceiling and records a non-converged terminal state / escalates to the operator (FR-014); it does not block indefinitely or loop unbounded.
- **Findings on a spec that is later abandoned/cancelled**: findings are recorded against the spec and persist with it (records preserve history; they are not deleted because the spec was abandoned).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST fire a cross-model audit-barrage over a spec **automatically** at a defined definition-time checkpoint, with no manual barrage invocation required.
- **FR-002**: The barrage MUST **attempt all configured model families in parallel** (for genetic diversity in failure modes), reusing the SAME barrage mechanism (render + fire + triage) as the implementation-phase governance rather than a parallel, separately-maintained implementation. A run with **≥1 covering family** (FR-008: liftable AND exit-0) is a valid, successful audit recorded with honest coverage (FR-008); one covering family is the floor (below two, the cross-model-agreement signal is simply unavailable — not an error). Zero covering families is an outage (fail loud, FR-005) — even when some families emitted bytes but crashed (non-zero exit), since a crash-after-banner family is liftable but does not cover the run.
- **FR-003**: The system MUST flag **cross-model agreement** (two or more model families flagging the same root cause) as a **confidence** annotation (`cross-model-agreed` vs `single-model`), distinguishable from single-model findings. Confidence is **orthogonal to severity**: a finding has both a confidence label (agreement) and a severity (`blocking`/`high`/`medium`/`low`/`informational`). The convergence gate (FR-010) counts **severity**, never confidence — the two axes MUST NOT be conflated. **Same root cause** is determined by clustering: two findings cluster when their headings share a **≥12-character substring** (case-insensitive, punctuation-stripped) **OR** they cite a **shared repo-relative path token**; clustering is transitive, and a cluster spanning ≥2 families is `cross-model-agreed`. **Known limitation for single-artifact specs**: the as-built path-token branch clusters on any shared repo-relative path token, so when the audit unit is one file (the usual spec-governance case — every finding cites the same `spec.md`) the path-token branch over-clusters (it would collapse findings that merely point at the file under audit). For single-file specs, therefore, the **heading-substring** branch is the load-bearing agreement signal; the headline cross-model signal must mean "two families independently flagged the same root cause," not "two families both pointed at the file under audit." But heading-substring carries the complementary failure mode: model-authored headings are free text, so two families flagging the same contradiction with differently-worded headings (no ≥12-character shared substring) will **under-cluster** — a real cross-model agreement silently recorded as two `single-model` findings. On single-file specs cross-model agreement clustering is therefore **best-effort in both directions** (path-token over-clusters, heading-substring under-clusters), and a `single-model` HIGH-severity finding MUST NOT be deprioritized on the assumption that a genuine issue "would have clustered."
- **FR-004**: The system MUST surface findings attributed per model family and route them into a triage workflow in which each finding receives an explicit, durably-recorded disposition.
- **FR-005**: The system MUST **fail loud when no audit capability is available** — it MUST NOT silently skip governance or report an ungoverned spec as governed (no-fallbacks principle). The "available" determination is **coverage-based** (FR-008/AUDIT-20260607-42): a run with **zero COVERING families** is an outage and fails loud, even when one or more families emitted bytes but crashed (non-zero exit) or timed out. A crash-after-banner run that produced bytes does NOT count as audit capability — counting it as such is exactly the silent-skip this FR forbids.
- **FR-006**: The system MUST use **stack-control's own audit-barrage capability AND audit protocol** — the audit protocol being the iterative convergence criterion (FR-010) plus the finding state machine / triage discipline, not just the one-shot barrage primitive. **Both are now vendored in-package in stack-control** (`multi/migrate-audit-barrage`, pulled forward 2026-06-07 per operator directive that dw-lifecycle is not an allowed dependency): the barrage verbs dispatch through the bundled `stackctl`; the gate imports the convergence criterion from stack-control's own source. **There is NO dependency on dw-lifecycle** (no import, no shell-out, no `requires`). The system MUST still fail loud if the barrage capability or model families are absent (FR-005). The barrage + protocol are NOT reimplemented separately for spec governance — the same verbs serve both definition-time and implementation-time governance (one capability, two phases).
- **FR-007**: Findings MUST be recorded in a durable, **append-only** per-feature audit-log, grouped by barrage run (one lift section per run, FR-009). Each entry carries a disposition — `open` → `fixed-<sha>` / `acknowledged-<reason>` / **`acknowledged-slush-pile-<date>`** (the binned-but-not-fixed state, FR-015) — that is **per-entry state set at triage** (FR-004); append-only means a dispositioned entry **keeps its disposition** across later spec revisions. There is **NO automatic cross-run finding-matching and NO disposition inheritance**: a re-barrage produces a **new lift section**, and a persisting defect **re-surfaces as a NEW `open` entry** to be triaged again (a `fixed-<sha>` defect that re-surfaces means the fix did not hold and is triaged anew; its earlier `fixed-<sha>` entry stays dispositioned). "Distinguish still-open from already-dispositioned" (SC-004) is the **literal `Status:` line on each entry**, not a similarity heuristic — which is exactly what makes the gate's literal-`Status:` counts reliable. The gate consumes these entries per the convergence rule (FR-010): the cross-run blocking open-set is a literal-`Status:` UNION over the entries (not similarity matching), and slushed entries (FR-015) remain in the record, re-openable only via a manual, out-of-loop burn-down.
- **FR-008**: When some but not all model families are available, the barrage MUST proceed with the available families AND **record reduced coverage** — degraded coverage must never be presented as full coverage. Two distinct predicates govern (AUDIT-20260607-42), and they must not be conflated:
  - **Liftability** — a model family's output is **liftable** when it **emitted ≥1 byte of stdout and incurred no spawn failure**. Liftability governs ONLY what the lift step extracts: every liftable family's `.md` is read for findings (a non-zero-exit family that emitted findings is still lifted whenever the run has coverage — liftable output is never discarded). No parseability claim is made; the lift simply extracts 0 findings from a non-finding blob.
  - **Coverage** — a model family **COVERS** a run when it is liftable **AND exited 0** (ran to completion; this also excludes timeout and spawn-failure sentinels). Coverage — NOT liftability — counts toward the reduced-coverage record; the fail-loud outage (FR-005) is the **zero-coverage** case. A crash-after-emitting-bytes family is liftable but NOT covering, so it can never by itself make a run governed-clean — closing the hole where a banner-then-`exit 1` family masqueraded as a clean covering run in the single-family floor (FR-005/US3/SC-003). The coverage bar is byte-presence + no spawn/timeout error + exit-0; it does NOT require output that parses into the finding schema.
- **FR-009**: A barrage run MUST be recorded even when it produces zero findings (a clean result is an outcome, not a skipped step), and a re-run MUST NOT be pre-emptively skipped as a presumed no-op (empty revisions beat missed changes).
- **FR-010** *(promotion gate — audit-protocol convergence; **THE canonical convergence rule — every other mention in this spec references here, none re-derives it**)*: Spec-governance MUST gate progression to the next Spec Kit step on the **ported audit-protocol convergence criterion**, counting open-finding **severity** (FR-003 — never confidence). The gate graduates a spec for a checkpoint **only when BOTH** of the following hold; until they do, the spec MUST NOT graduate (a machine-checkable gate mirroring `impl/execution-engine` FR-030):

  1. **The dampener is engaged** (the *per-run consecutive-quiet verdict*). The dampener engages when **EITHER** (a) one barrage iteration yields **0 open HIGH (or BLOCKING) and 0 open MEDIUM**, **OR** (b) the **two consecutive** most-recent iterations each yield **0 open HIGH (or BLOCKING)** (the **two-consecutive-0-HIGH window**). The dampener IS the `dampened = (branch a) OR (branch b)` predicate; the two-consecutive window is *one of the two ways it engages, not the definition of it* — so branch (a) single-run-clean graduation is fully reachable (a clean barrage of a fixed spec engages immediately, no second run needed). The branches are **intentionally asymmetric** (it is the ported protocol): branch (a) requires 0 open MEDIUM in that one run; branch (b) does not itself require 0 MEDIUM.

  2. **The cross-run blocking open-set is empty** (the *cross-run union*). This is the **UNION of un-dispositioned HIGH/BLOCKING findings across ALL of the checkpoint's recorded runs** — a literal `Status:`-line scan over every lift section, NOT only the most-recent run (and NOT a similarity / cross-run matching heuristic; cross-run finding-matching was deliberately rejected as fiction, FR-007). A HIGH/BLOCKING finding surfaced `open` in an earlier run, then stochastically not re-flagged later, **still blocks graduation until it is dispositioned** (this is what makes SC-006 literally true). Only the blocking open-set is unioned; the dampener verdict (part 1) stays per-run.

  **No graduation state carries open MEDIUM findings.** Branch (a) graduates with 0 open MEDIUM by its own condition. Branch (b) is made safe by the **slush step** (FR-015): it runs **automatically, before the gate evaluates, on every protocol pass** (chain: render → barrage → lift → **slush** → gate), and the moment the dampener engages it bins **all** still-open MEDIUM/LOW findings across the checkpoint's runs to `acknowledged-slush-pile-<date>` (not fixed, not open; re-openable only via a manual, out-of-loop burn-down, FR-015). So by the time branch (b) graduation is evaluated, no open MEDIUM remains anywhere in the checkpoint. **HIGH/BLOCKING are NEVER slushed**: a recorded open HIGH/BLOCKING satisfies neither branch, so it resets the dampener and re-surfaces as next work; it is cleared only by a recorded **fix** (`fixed-<sha>`) or a recorded **acknowledgment with a substantive reason** (the same evidentiary bar as an override) — never silently (the finding state machine enforces the recorded reason).

  An **iteration** is one recorded barrage run (FR-009); "**consecutive**" means the last runs **for the same checkpoint** (per-checkpoint scoping, FR-011/FR-014), regardless of whether the spec text changed between them — an inter-iteration edit does NOT reset the count; two-consecutive-quiet is a stability heuristic, not a determinism proof. An **override** is an operator action recorded with a **mandatory reason** (supplied via `GOVERN_OVERRIDE` / `--override "<reason>"`, recorded in the run verdict as `override.recorded` + `reason`; no reason → no override). An override is **scoped** to the spec (feature) + checkpoint it applied to (auditable per spec/checkpoint); the explicit `--override` flag is **preferred** for a single scoped bypass, while a persistent exported `GOVERN_OVERRIDE` would apply to subsequent runs and is a **known hazard** operators must avoid. This makes spec-governance an **iterative loop**: barrage → triage/fix the spec → re-barrage → … until convergence.
- **FR-011** *(hook point)*: The spec-barrage MUST fire at **`after_clarify`** (the spec is decision-complete then) and MUST be **configurable to also fire at `after_plan`**. `after_specify` is intentionally **out of scope** — a spec at that point may still carry intentional unresolved-clarification placeholders, which would generate noise; only `after_clarify` and `after_plan` are wired checkpoints with defined artifact sets (FR-013). `after_plan` is **off by default** (the default is `after_clarify` only); it is enabled per project by marking its hook non-skipped in the extension manifest (`extension.yml` `hooks.after_plan`) and is fed the plan via `GOVERN_PLAN_PATH` at fire time. When **both** checkpoints are enabled, each runs an **independent convergence loop** (FR-014), and the gate is **scoped per-checkpoint** — it evaluates only that checkpoint's recorded runs (via the checkpoint filter, FR-010). So passing the `after_clarify` gate is **durable**: `after_plan`, evaluated in its own loop, does NOT re-open it. A spec-level issue that re-surfaces at `after_plan` is simply a **new `open` finding in the `after_plan` loop**, triaged there (FR-007). There is **NO cross-checkpoint disposition inheritance** — each checkpoint's open-set is its own recorded runs.
- **FR-012** *(delivery mechanism)*: Spec-governance MUST be delivered as a **Spec Kit governance extension with hooks** (the hook mechanism is **mandatory**), mirroring the founding `deskwork-governance` `after_implement` extension, so it fires **universally** — whether the operator drives the front-door `define`/`extend` skills or raw `/speckit-*` commands. It MUST NOT be folded into the front-door skills only; the front-door skills are callers that benefit from the universal hook, never an alternative delivery surface.
- **FR-013**: The spec-barrage operates on the **spec** by default (the `after_clarify` checkpoint's artifact set); when the `after_plan` checkpoint is enabled (FR-011), that checkpoint's loop covers the **plan** (additive — spec + plan). No other artifacts are in scope.
- **FR-014** *(iterative convergence + bounded termination)*: The system MUST support repeated barrage iterations, each recorded as its own append-only lift section (FR-007), until the convergence criterion (FR-010) is met or an override is recorded. Each iteration MUST be a recorded run (FR-009). The loop MUST be **bounded** — if convergence is not reached after a configured iteration ceiling, the system MUST surface **non-convergence** (escalate to the operator / record a non-converged terminal state) rather than loop forever. **Forward path from `non-converged`**: the spec does NOT auto-graduate. Two recorded resolutions: (a) the operator records an **override** at the ceiling — the same mandatory-reason, scoped discipline as a finding-level override (FR-010) — which graduates the spec despite non-convergence; or (b) the operator fixes the findings and **continues**, raising the ceiling for the continued attempt via `--ceiling` / `GOVERN_CEILING` when more iterations are needed (an explicit, recorded flag value — bounded and auditable, not a silent re-run). Continuing after a fix is just **more recorded runs**, evaluated by the convergence rule (FR-010) over the latest runs — there is no separate window-reset mechanism because none is needed; and per FR-010's cross-run union, a clean re-barrage **alone** does not graduate while an earlier run still carries an un-dispositioned open HIGH/BLOCKING (the operator must disposition the earlier HIGH — this is the intended SC-006 behavior: a HIGH is explicitly resolved, not flickered out by a stochastic clean run). Absent an override, a `non-converged` spec does NOT graduate. The iteration ceiling is **per-checkpoint** (FR-011): each enabled checkpoint has its own loop and its own ceiling (**default 5**, configurable via `--ceiling` / `GOVERN_CEILING`), counted over that checkpoint's recorded runs only — not a single global budget shared across checkpoints. **Spec-governance governs; it does NOT auto-edit the spec** — the "fix" step of the loop is the **author's** act (the agent/operator driving the define/clarify cycle). "Unattended" means the **gate** evaluates and bounds the loop without an operator present (recording non-convergence at the ceiling), not that the spec is machine-edited. This keeps the loop safe under the unattended/all-night directive.
- **FR-015** *(slush pile — the slush MECHANISM; FR-010 owns *when* it fires)*: When the dampener is engaged (the convergence rule, FR-010, for *when*), the system MUST **slush** the residual MEDIUM/LOW findings — flip their disposition to `acknowledged-slush-pile-<date>` (a third state, distinct from `open` and `fixed-<sha>`: acknowledged-as-not-fixed). **Slush scope is ALL still-open MED/LOW across the checkpoint's recorded runs**, not only the most-recent one (AUDIT-20260607-47): an earlier 0-HIGH run's MEDIUM — never slushed when it fired, because the dampener was not yet engaged then — is binned at convergence too, so "slush any remaining" is literally true and no open MEDIUM survives into graduation (SC-007). The flip is confined to the engaged checkpoint's runs (FR-011), never crossing checkpoint boundaries. The slush step is **automatic and runs on every protocol pass, before the convergence gate evaluates** (the chain is render → barrage → lift → slush → gate); it is **not** an optional or separately-invoked operation in the governed loop. It is a **no-op until the dampener engages** (it refuses to slush otherwise — so an early, still-noisy pass slushes nothing), which is why the gate that evaluates immediately afterward never sees the residual MEDIUMs as open. Slushing **MUST refuse while the dampener is not engaged** (real HIGH bugs still surfacing must not be binned), and MUST **NEVER** slush HIGH/BLOCKING findings. Slushing is what lets a stable-0-HIGH spec graduate without an unbounded grind on residual MEDIUMs. The slush MUST be reversible, but the reversal happens **outside the governed auto-convergence loop**: a **burn-down** is a **deliberate, manual remediation pass** (a separate `slush-findings --burn-down` invocation, never wired into the protocol chain) that re-opens slush-pile findings so the operator/agent can address them directly as a standalone remediation activity ("address all findings, but bin the smaller items once two consecutive audits had 0 HIGH" — and revisit the pile when we choose). **Burn-down is NOT fed back through the governed loop.** Because the auto-slush step is automatic-on-engage-before-gate on every protocol pass (and the dampener stays engaged while HIGHs are clear), any findings re-opened inside the governed loop would simply be re-slushed by the next pass before the gate sees them — so burn-down's re-opened findings are remediated out-of-loop (fixed directly, or re-slushed by the operator), never round-tripped through auto-convergence. There is **no stateful "burn-down suppresses auto-slush" mode**. Slushing (the auto step) is gated on the same per-checkpoint dampener as the gate (FR-011).

### Key Entities *(include if feature involves data)*

- **Spec-barrage run**: one automatic invocation of the cross-model barrage over a spec at a checkpoint; carries the model families exercised, coverage (full/reduced), timestamp, and the produced findings. Recorded durably.
- **Finding**: a single issue raised by the barrage; carries the originating model family (or families, when agreement), a **confidence** label (`cross-model-agreed` when ≥2 families flag the same root cause — clustered by ≥12-char heading overlap or shared path token, FR-003 — else `single-model`), an **orthogonal severity** (`blocking`/`high`/`medium`/`low`/`informational` — the axis the FR-010 gate counts), the cited spec location, and a disposition (open / fixed / acknowledged). Confidence and severity are independent: a finding can be `single-model` + HIGH-severity (still gate-blocking) or `cross-model-agreed` + low-severity.
- **Checkpoint (hook point)**: the definition-time moment at which the barrage fires — `after_clarify` (spec) and/or `after_plan` (spec+plan); `after_specify` is out of scope (FR-011, no wired artifact set). Each enabled checkpoint owns an **independent convergence loop** with its **own iteration ceiling** (FR-011/FR-014) over its own artifact set (FR-013); a passed checkpoint gate is durable across other checkpoints' runs.
- **Audit capability**: stack-control's **own** (in-package) barrage primitives + the available model families. A family **COVERS** a run when it emits ≥1 byte of stdout, incurs no spawn/timeout error, **AND exits 0** (FR-008/AUDIT-20260607-42 — byte-presence + no error + ran-to-completion, no parseability claim); zero COVERING = outage (fail loud, FR-005). A family's output is separately **liftable** when bytes>0 + no spawn failure — liftability governs only what the lift extracts (a non-zero-exit family is lifted but does not cover). Coverage recording (FR-008) counts COVERING families. No dw-lifecycle dependency (FR-006).
- **Audit protocol**: stack-control's **own** convergence discipline (vendored via `multi/migrate-audit-barrage`) — the iteration loop, the convergence criterion (FR-010 — the canonical rule), the finding state machine, the **slush pile** (FR-015), and the bounded per-checkpoint iteration ceiling (FR-014). The protocol pass runs render → barrage → lift → **slush** → **gate** in that order (per FR-010/FR-015, the slush bins all residual MED/LOW before the gate evaluates, so graduation never carries open MEDIUM findings). Distinct from the barrage primitive (which produces findings in one pass); the protocol governs **how many passes run and when the gate opens** — the mechanic itself is defined once in FR-010.
- **Governance run (iteration set)**: the sequence of barrage iterations for one spec graduation attempt; carries each iteration's run, the convergence outcome (converged / overridden / non-converged), and the iteration count vs. ceiling.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A spec that reaches the defined checkpoint has a barrage run recorded **with zero manual barrage invocations** — automatic governance is observable in the run record.
- **SC-002**: When ≥2 model families flag the same root cause, that finding is labeled **`cross-model-agreed`** (the HIGH-confidence signal) and is distinguishable from `single-model` findings in the output. (Confidence is an annotation, orthogonal to the severity the gate counts — FR-003.)
- **SC-003**: When no audit capability is available — including a zero-COVERING run where families emitted bytes but all crashed (non-zero exit) or timed out (AUDIT-20260607-42) — the flow fails loud with an actionable message and the spec is NOT recorded as governed. **Zero silent skips** across all runs; a crash-after-banner outage is never reported as a clean governed run.
- **SC-004**: Across a spec revised at least twice, prior runs' dispositions are preserved in their own lift sections (append-only — no disposition loss); each re-barrage records its findings in its **own** lift section (no cross-run finding-matching or disposition inheritance), and a still-present defect re-appears as a new `open` entry rather than being silently carried as fixed. (The gate then evaluates the cross-run union of un-dispositioned HIGH/BLOCKING by literal `Status:`, FR-010 — so the new `open` entry blocks until dispositioned, while the prior run's own entries keep their recorded dispositions.)
- **SC-005**: Spec-phase findings and implementation-phase findings appear in the **same format AND the same store** — a single in-stack-control governance surface, two phases. (Now literally true post-`multi/migrate-audit-barrage`: both phases use stack-control's own audit-barrage + the same per-feature audit-log; there is no cross-plugin store and no dw-lifecycle coupling to preclude it.)
- **SC-006** *(measurable via the deterministic gate, not the stochastic barrage)*: Once a seeded self-contradiction is **surfaced as an open HIGH+/BLOCKING finding** in **any** of the checkpoint's recorded runs, the convergence gate **does NOT graduate the spec until that finding is dispositioned** (fixed, acknowledged, or recorded-override) — **even if later runs stochastically fail to re-flag it**. This is the *nothing-serious-slips-through* guarantee, honored by the cross-run blocking open-set per the convergence rule (FR-010): a flickered-out earlier HIGH still counts, so the gate stays blocked until the operator dispositions it. It is checkable and deterministic (the gate counts open findings by literal `Status:`, so its behavior is deterministic even though the barrage is stochastic). Detection itself is **not** part of the guarantee: the cross-model battery surfaces the contradiction with greater odds than any single model, but a contradiction the barrage never records cannot be required to be dispositioned. (Catches the `specs/002` "author introduced 3 contradictions" failure mode once the barrage flags it.)
- **SC-007**: A spec cannot graduate to the next Spec Kit step until the convergence criterion (FR-010) is met or an override is recorded — verifiable in the run record. The two graduation guarantees both follow from FR-010: (1) **no spec graduates while ANY of the checkpoint's recorded runs carries an un-dispositioned open HIGH+/BLOCKING finding** (the cross-run union), absent a recorded override; and (2) **no spec graduates carrying any open MEDIUM finding** — because the slush step (FR-015) bins **all** still-open MED/LOW across the checkpoint's runs before the gate evaluates, leaving none open anywhere at graduation (slushed = not fixed, not open, re-openable only via a manual, out-of-loop burn-down — never silently dropped). (Guarantee (2) is a clean absolute, restored by AUDIT-20260607-47's slush-all behavior: there is no "gating run" scoping caveat because no earlier-run MEDIUM survives the convergence slush.)
- **SC-008**: An iterative governance run that cannot converge terminates in a recorded **non-converged** state within the configured iteration ceiling — it never loops unbounded (verifiable: the run record shows either convergence, a recorded override, or a non-converged terminal state).

## Assumptions

- **Audit-barrage + audit-protocol source**: this feature uses stack-control's **own** barrage capability AND audit protocol (convergence criterion + finding state machine), **vendored in-package** via `multi/migrate-audit-barrage` (pulled forward 2026-06-07). It does NOT depend on dw-lifecycle and does NOT build a second implementation — the same in-stack-control verbs serve both definition-time and implementation-time governance. Fail-loud-if-absent (the barrage capability / model families) remains the contract (FR-005).
- **Findings home**: findings are recorded in the per-feature audit-log durable store used by the implementation-phase governance (stack-control's own), carrying the spec checkpoint context — not a separate spec-only artifact. Post-migration this is a single in-stack-control store shared by both phases, so SC-005 (one surface) is literally satisfied, not just format-compatible.
- **Synchronous / iterative (resolved by FR-010)**: the gate is an iterative convergence loop, so the barrage result must be available before graduation — the gate waits for convergence. **Unattended applies to the gate, not the fix**: the gate can evaluate and record bounded non-convergence without an operator present (FR-014); it never auto-edits the spec. The **fix** between barrages is always an authoring act by the agent/operator under the normal spec-editing contract (which may itself be agent-driven via fresh-context per-finding dispatch), performed before a fresh run. The loop remains bounded (FR-014) for the all-night case.
- **Artifact scope (default for FR-013)**: the spec is always in scope; the plan is included only if a `after_plan` checkpoint is selected (FR-011).
- **Naming / branch**: built on the shared `feature/pluggable-lifecycle-providers` branch under `specs/004-spec-governance/` (one branch per worktree; spec dir is flat creation-order, the `design/spec-governance` codename is the identity).
- **Sequence**: this is the second feature in the resequenced design-phase block (after `design/insight-capture`), per `stack-control-roadmap.md`. It is independently shippable.
