Deskwork architectural thesis.

Read this before touching any deskwork code. This file exists because the thesis was implicit in the project's design but not written down anywhere discoverable, which let the wrong architecture creep into the studio's review surface and waste an angry session correcting it.

The canonical source

"You never leave the agent for a static tool. You talk to it. It does the work." The agent's toolkit isn't a static configuration you set up once; it evolves, under your direction, in response to what the work actually needs.

The blog post is the canonical source. Everything below derives from it.

What the thesis means

The agent (Claude, in a Claude Code session) is the primary tool. Every other piece of deskwork — CLI verbs, the studio, doctor rules, sidecars, the calendar — exists to be read, edited, and orchestrated by that agent. Static tools (opaque code the agent can't see or evolve, server endpoints that hide multi-step logic, UI that takes business-logic actions without surfacing what it's doing) are what the thesis rejects.

B · outline open, note expanded inline

Notes on the pattern. No drawers, no FAB, no modal sheets. The outline lives at the top of the article as a collapsible <details> element — tap to expand into a section list with note-count pips per section. Each section link jumps the operator into the article (anchor scroll), then the operator can re-collapse the outline.

Margin notes are spatially adjacent to the text they reference. Each highlighted run carries a small note-count badge floating just past its right edge. Tap the highlight (or the badge) and the note expands inline immediately AFTER the paragraph — exactly where the operator's eye is — with Resolve / Edit / Delete actions on the card itself. The note is dismissed by the same tap, by tapping anywhere else, or by Resolve.

The persistent bottom action bar is the only permanent secondary chrome — three decision buttons (Approve / Iterate / Reject) and a help affordance. 56px tall, thumb-reachable, always visible. No mode-switching, no hidden state, no "where did the buttons go" moments.

This direction trades a more opinionated information architecture for a more legible reading experience. The note-count chips need careful tuning to not overpower the prose; the inline-note callout needs a max-height/overflow to avoid hijacking the article when there are many notes on one paragraph.
