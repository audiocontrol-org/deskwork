B · both trays pulled out · top: back / stamp / mode / save · bottom: format / notes / outline / discard · trays auto-hide after ~2s of typing

PatternThe editor IS the page. No top strip, no bottom tab bar, no sheets at rest. Two thin pull-tabs sit on the top and bottom edges (above the keyboard). Pulling one out reveals a tray that auto-hides after ~2s of typing. The keyboard accessory bar (B / I / code / H1 / H2 / link / list / quote) is the ONLY persistent UI when typing — the markdown helpers, always at the thumb.

Top trayPull down to reveal: back glyph, stage stamp (rotated, green when editing), Source/Preview pill, Save action button (stamp-green when dirty, faded when clean). The save state is visible only when the operator has pulled the tray — but a tiny corner-of-eye indicator (the green-ringed pull-tab when dirty, neutral when clean) gives a lower-fidelity signal at all times.

Bottom trayPull up to reveal: Format / Notes / Outline / Discard. Format opens a sheet of markdown press-check keys (same as Mockup 2). Notes opens the same Notes sheet from review. Outline opens the table of contents. Discard exits without saving. All four trays slide up over the keyboard accessory; the accessory remains the bottom-most thing.

Word countA faded mono label in the top-right corner — "1,247 words" — is visible at all times. Editorial language preserved: literal press-check counter, not a chrome element.

Why this is right (and the trade-off)The current edit toolbar (#239) crams 9 chrome elements into a row that wraps on phone. This direction goes past minimal toward absent — the chrome IS hidden by default, revealed by pull-gesture. Best for sustained writing sessions where the operator just wants to type. The cost: discoverability. Operators have to learn that pull-tabs reveal trays. Mitigation: subtle pulse-once animation on the pull-tabs at edit-mode entry, so the tabs announce themselves the first time. The most opinionated direction; trades familiarity for focus.
