This document catalogs the split-bar's contents in every context the studio registers today — 9 distinct contexts across 7 surfaces + 2 modal variants — and identifies where the pattern strains. One bar pattern, one universal nav region, three contextual cells maximum. If anything can't fit, that's signal the pattern is wrong. If everything fits with clarity, we ship it.

§ 1 Bar anatomy

One bar across every studio surface. 56px tall. 1px paper-3 border-top. 3-cell nav region on the left (fixed: Desk · Short · Manual). Kraft hairline divider at the boundary — visible all the way through the bar's height. 1–3 contextual cells on the right (surface-specific). All cells use mono caps at 0.5rem with 0.16em tracking; all glyphs use Newsreader italic display at 1rem.

⌗Desk ⊟Short §Manual

§ 5 Compression decisions & escape hatches

Decision 1 · Entry-review review-mode loses the Scrapbook tab. Today's v0.18 mobile-bar has four tabs in review mode: Outline · Notes · Scrapbook · Actions. Split-bar caps contextual at 3 (with 3 fixed nav). Scrapbook compresses into the Actions sheet as a section labeled Folio, exactly per the Scrapbook-3 mockup that was explored for the dashboard's row-affordance scrapbook follow-up. The folio count moves from a bar badge to a sub-section badge inside the Actions sheet. Cost: one indirection step (open Actions → see folio count). Benefit: every studio surface uses the same bar shape.
Decision 2 · Verb cells use one-tap clipboard-copy; sheet-tab cells open sheets. Two cell modes in the contextual region: verb cells (iter / apprv / canc / save / compose) fire /deskwork:<verb> directly on tap — no sheet, no confirmation, the clipboard-copy IS the act per THESIS Consequence 2. Sheet-tab cells (outl / notes / acts / fmt / filter / search / sect / open) open a Phase 2.1 createSlideUpSheet() instance. Operators learn the distinction by color: verb cells use accent colors, sheet-tab cells use faded glyphs (with active-state red-pencil top rule on the active sheet's cell).
Decision 3 · Scrapbook viewer's nav highlight follows the parent entry's tab. When opened from a longform entry (Desk-rooted), the Desk tab highlights. When opened from a shortform workflow (Short-rooted), the Short tab highlights. The scrapbook viewer doesn't get its own nav slot — folio is "context for the entry," not a peer destination. Tapping the highlighted tab while in the scrapbook viewer returns to the parent entry's review surface (not all the way to Desk/Short index). Maintains the "tap active = return to that tree's root, where root = whichever entry you were in" mental model.
Decision 4 · Escape hatch — when a surface needs >3 contextual cells. If a future surface needs a 4th contextual cell, the third cell becomes a "More" tab opening a sheet of secondary verbs. This is the pattern the dashboard's row-affordance picked (Row-4 ⋮ menu for block / induct / etc.). It avoids the temptation to widen the bar past 6 cells (which would shrink cells below the 65px tap-target floor). Current state: no surface needs this yet. Adding it later is non-breaking — the third cell semantics doesn't change for surfaces that already use it for an actual verb.

§ 6 Open questions for operator pick

Refinements the mockup pass surfaces but doesn't unilaterally answer. Each needs an explicit operator pick before implementation.

Question 1 · the Scrapbook compression Confirm: Scrapbook tab moves into the Actions sheet as a "Folio" section in entry-review review-mode? Or would you rather keep Scrapbook visible and drop a different tab (e.g., Outline — operators can re-enable via a sheet from Notes)? The Scrapbook-3 precedent argues for moving Folio into Actions (folio = "context for the decision"); the cost is one tap of indirection plus the loss of an at-a-glance Folio badge.
Question 2 · the Compose route Compose currently lives as a floating chip on the dashboard (v0.19 shipped). Should it: (a) stay floating + appear in the bar as the contextual cell only on dashboard, (b) appear in the bar on every surface (Compose anywhere → opens compose sheet with platform/parent-entry pickers), or (c) stay floating-only and the bar's contextual region on dashboard renders something else (e.g., a stage-filter)? This mockup assumes (a) — Compose as bar cell on Dashboard only — but the floating Compose chip is retired in that case (only one Compose entry-point should exist).
Question 3 · Shortform desk's contextual cell This mockup proposes Filter as the shortform desk's one contextual cell — opens a platform multi-select sheet. Alternative: + New shortform (compose-style, opens a sheet asking platform + channel + parent entry). The desk's job is to navigate to existing workflows + start new ones; Filter helps navigate, +New helps create. Probably want both eventually; pick which is primary.
Question 4 · Content tree as a peer or sub-surface Should the Content tree (/dev/content/<site>) be a sub-surface of Desk (its nav active = Desk, accessed via a kicker breadcrumb or sub-link) — or a peer destination (4th nav tab, requiring the nav region to widen to 4 cells)? This mockup assumes sub-surface. Adding a 4th nav tab would shrink each nav cell from 65px to ~56px and squeeze the contextual region accordingly. Recommendation: nest under Desk until operator feedback says otherwise.
Question 5 · Manual's two contextual cells Manual carries Sect. (table of contents sheet) and Search (in-document search). Sect. is high-value if the manual gets long; Search is high-value at any length. Two cells fits — confirm both? Or compress to one ("Index" → sheet with TOC + search combined)? Manual hasn't been built yet so this is a forward-looking spec, not a current refactor.
Question 6 · Studio index — bare nav or "discover" cell? The Studio index at /dev/ renders a directory of surfaces. Should the bar at that surface have (a) no contextual cells (just 3 nav, ~130px each — luxurious tap targets), or (b) a single "Discover" cell that opens a surface-list sheet (redundant with the page content)? Recommendation: (a) — the index page IS the directory; the bar doesn't need a duplicate.

§ 7 Implementation path

What the implementer dispatches in Task 2.2.5 + 2.2.6 should look like, assuming Cross-bar 2 + the decisions above.

Server template extraction Task 2.2.5's renderMobileBar(opts: MobileBarOptions) becomes: 
renderMobileBar({
  navActive: 'desk' | 'shortform' | 'manual',
  contextual: Array<{
    kind: 'verb' | 'sheet-tab',
    glyph: string,        // 1-char display italic
    label: string,        // ≤5 chars
    accent?: 'iterate' | 'approve' | 'cancel' | 'save' | 'compose' | null,
    badge?: { count: number, tone: 'red' | 'kraft' | 'green' },
    action: { type: 'copy-cmd', cmd: string }
          | { type: 'open-sheet', sheet: string }
          | { type: 'direct', handler: string }
  }>  // max 3
})
Two concrete consumers from day one: entry-review/mobile-bar.ts (existing) refactored to call this; shortform-review as the new consumer added in Task 2.2.6. Future consumers (scrapbook, manual, content tree) slot in by passing their contextual array.
CSS extraction Move the bar's CSS rules to plugins/deskwork-studio/public/css/mobile-shell.css (new file). Existing editorial-review.css rules at :3703 ([data-review-ui="longform"] .er-mobile-bar) get rescoped to drop the [data-review-ui="longform"] attribute — the bar is universal now, not longform-only. Compose-chip CSS in editorial-studio.css retires (the chip moves into the bar as a cell, per Decision 2).
Phase 2.1 primitive reuse Each sheet-tab cell instantiates a createSlideUpSheet() via the existing controller. The sheet host is shared across surfaces — the bar's data-mobile-sheet attribute on the active tab routes the open call. No new client-side primitives required.
