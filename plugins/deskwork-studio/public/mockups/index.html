Mobile design exploration

Mobile-first design directions for the studio's two surfaces.

Two phone-shaped surfaces, two design problems. The review surface shipped a mobile-first rebuild from these mockups (Mockup 1 was picked); it now reads cleanly on phone and the operator can review press-check style with thumb-reach chrome. The editor is the next surface to redesign — currently a compressed desktop layout (split-pane crushed, action row wraps to two lines, mode tabs invisible to the thumb). The three editor directions below honor the same press-check vocabulary already established and explore three different relationships between chrome and the writing surface.

Shared design principles

  • Content first. Article (or editor) fills the viewport. Chrome is bottom-anchored, sheet-based, or content-inline — never a side-rail.
  • Thumb reach. Frequent actions (decisions, save, format) sit in the lower half where the thumb naturally lands.
  • One sheet at a time. Outline, notes, format are peers; opening one closes the others. No competing overlays.
  • Native idioms. Drag-handles to dismiss, swipe-down to close, tap-outside-to-close. Keyboard-accessory bar above the iOS keyboard for editor markdown shortcuts.
  • Editorial language preserved. Cream paper, red-pencil accents, JetBrains Mono kicker labels, Newsreader italic display, stage stamps with a hint of rotation — expressed as mobile-native components.
  • Per-action color accents. Approve / Save = stamp-green. Iterate / Source-Preview toggle = proof-blue. Reject = red-pencil. Cancel / Discard = faded. Same vocabulary across review surface and editor.

Dashboard row 1

Tap-anywhere + overflow menu

Weak Every secondary action is 2 taps (⋮ → action). Bulk operations slow.

Review surface · scrapbook follow-up Active

The shipped mobile review surface (Mockup 1, the bottom tab bar) has no affordance to open the scrapbook — the desktop drawer is hidden behind the fixed 64px bar on phone widths. Three directions for placement, each honoring the press-check vocabulary already shipped. Edit mode is unaffected (operator is writing, not browsing assets).

Scrapbook 1

Fourth tab

Bar grows to 4 tabs in review mode (Outline / Notes / Scrapbook / Actions). Tapping Scrapbook opens a sheet with a 2-column card grid; each card uses a format-specific abstract thumbnail (red↘blue gradient for images, paper-rule pattern for text, waveform for audio). Standalone viewer link in the sheet head.

Strong Predictable. Same idiom as existing tabs. Item count surfaces at-a-glance via badge.

Weak 97.5px per tab is denser than the 130px 3-tab layout. Dilutes the "thumb-reach for decisions" framing of the original Mockup 1.

Scrapbook 2

Folio peek strip

A ~36px kraft-warm strip wedges between the top chrome and the article body — notched left edge cut like a hanging-file folder tab, italic "Folio" label, item count, chip-thumbs previewing format diversity, and a viewer link. Tap expands the same sheet pattern. Bar stays 3 tabs.

Strong Most distinctive — a new visual artifact rather than another tab. The metaphor (article inside a folio with materials) is structurally true and reads instantly. Format chips give at-a-glance content diversity.

Weak +36px chrome above article. Notch + clip-path needs careful CSS. Empty-folio state needs handling.

Scrapbook 3

Actions sheet section

Zero new chrome. Existing Actions sheet gains a third section between Document and Decisions: Folio. Each item renders as a compact one-line row (swatch + filename + format chip + size). Standalone viewer link in the section header. Sheet grows to 70% height to accommodate.

Strong Most minimal — no new tab, no new strip, no new visual artifact. Semantic: folio is "context for the decision" and lives next to the decisions.

Weak Discoverability cost — the operator has to know Actions holds folio. Item count requires opening the sheet (Notes count badges at-a-glance; Folio doesn't, unless we add a second indicator).

Editor · three directions Mockup 2 shipped

The mobile editor rebuild is shipped (operator picked Mockup 2 / press-check tab bar; commits cdc184a + 5baa042). The original three editor options stay below for reference.

Editor 1

Keyboard accessory

Chrome surrenders to the editor. At rest: only a back glyph and a Save state pill (stamp-green when dirty, faded when clean). When typing, a 48px accessory bar of markdown shortcuts (H1 / B / I / code / link / list / quote) rides directly above the iOS keyboard. The page becomes a writing surface.

Strong Maximum viewport for the editor. Markdown shortcuts at the thumb. Save state visible at all times via a single colored pill.

Weak Less continuity with the review surface's bottom tab bar. Mode toggle (Source / Preview) hidden in the bottom strip when keyboard is down.

Open mockup →
Editor 2

Press-check tab bar

Same 64px bottom-tab-bar idiom as the review surface — three tabs with editor verbs (Format / Notes / Save). The Format tab opens a sheet of press-check markdown keys grouped by function. The Save tab glows stamp-green when the document is dirty. Mode (Source / Preview) is a strip segment in the top chrome. Stage stamp shifts to green to signal editing.

Strong Direct continuity with the review surface — operator's muscle memory transfers without recalibration. Notes sheet pattern is reused unchanged. Save state is loud and obvious.

Weak 96px top + 64px bottom = 160px of chrome at rest, less than Mockup 1. No keyboard accessory by default — markdown helpers are one tap deeper (Format sheet).

Open mockup →
Editor 3

Writing room

Distraction-free. No top strip, no bottom tab bar, no sheets at rest. Two thin pull-tabs sit on the top and bottom edges; pulling reveals trays that auto-hide after ~2s of typing. The keyboard accessory bar is the ONLY persistent UI when typing. A faded mono word count sits in the top-right corner. The most opinionated direction — closest to a Medium / iA Writer feel.

Strong The editor IS the page. Sustained writing sessions get nothing but text and shortcuts. Word count + persistent accessory honor editorial language without competing for attention.

Weak Discoverability — operators have to learn the pull-tab gesture. Save state is implicit unless the top tray is pulled. The trade-off: focus over familiarity.

Review surface · the three directions Mockup 1 shipped

The review surface rebuild is shipped (operator picked Mockup 1 on 2026-05-08; full implementation in commits 3e33ceb → d2fed62). The original three options stay available below for reference and for combining into the editor mockups.

Review 1

Bottom tab bar & sheets

A persistent 3-tab bar pinned to the bottom of the viewport. Tapping a tab reveals a sheet that slides up over the content. Outline, Notes, and Actions are peers, each in their own sheet. The "Apple Maps" pattern — predictable, native, always one tap away.

Strong Discoverability is permanent. Thumb-reach is excellent. Counts on tabs surface state at a glance.

Weak Permanent ~56px bottom chrome eats viewport. Three buttons may feel like clutter on the simplest entries.

Open mockup →
Review 2

Single FAB & radial menu

No permanent bottom bar. A single floating action button sits bottom-right; tapping it reveals a vertical stack of secondary actions (Outline, Notes, Approve, Iterate, Reject). Each opens its own sheet. A quieter chrome — one round button instead of a strip.

Strong Maximum content space. The button is a single, obvious entry point. The FAB itself is a recognizable pattern (Material, Twitter, Gmail).

Weak One-tap-deeper for everything. Counts (note count, etc.) need to surface inside the menu, not at-a-glance.

Open mockup →
Review 3

Inline reveal

No drawers. No FAB. The outline is a collapsible <details> at the top of the article; margin notes appear inline next to the paragraph they reference, behind a small per-paragraph chip; decision buttons live in a thin always-visible bottom bar. The most opinionated direction — content and notes are spatially related, not sheet-separated.

Strong Notes-near-text matches how operators actually read margin notes. No mode-switching, no stacked chrome, no ceremony. Each surface is "just there."

Weak Outline-at-top adds vertical preamble before the article begins. Inline note chips need careful design to not overpower the prose.
