The menu glyph — why three custom rules instead of ☰ The Unicode ☰ (TRIGRAM FOR HEAVEN) is the generic web "hamburger." Three CSS-rendered rules at varying widths (14px / 10px / 14px) read as a press-check editor's contents-mark — a subtle bibliographic flourish rather than a stamp-marked generic affordance. Distinct from the row-overflow ⋮ (used elsewhere) by orientation (horizontal vs vertical) so the two affordances don't conflate. WCAG: rules at 1.5px solid var(--er-ink) on var(--er-paper) = 16.1:1 contrast, well above the 3:1 non-text minimum.

PROPOSED ADDITION · DESIGN-STANDARDS.md (after § Surfaces, before § Mobile vs desktop deltas)

§ Studio navigation model

The studio uses a star navigation model. The Desk (/dev/editorial-studio) is the hub. Every other surface (entry-review, shortform review, scrapbook viewer, content tree, manual, future surfaces) is a leaf reached by navigating from the Desk. There is no bar nav region; there is no destination picker; there is no top-level tab bar.

The two universal masthead affordances

Every studio surface MUST carry these two masthead affordances at the breakpoints they're shipped to (mobile per this work; desktop in a future feature branch):

  • "← Desk" back-link — leftmost masthead position. Mono ← + italic-display Desk. Proof-blue, 1px underline. Tap navigates to /dev/editorial-studio. Absent only on the Desk itself (you're already home).
  • Menu glyph — rightmost masthead position. Three CSS-rendered horizontal rules (14px / 10px / 14px) inside a 30×30 paper-bordered square. Tap opens the studio menu sheet (Phase 2.1 createSlideUpSheet) with cross-cutting actions: Manual / Keyboard shortcuts / Configure / File an issue / About / future items. Present on every surface including the Desk.

The Desk's hub role

The Desk is the entry-point to every reachable destination. Its information architecture grows to absorb destinations that would otherwise be top-level navigation entries:

  • Longform pipeline tiles (Ideas → Cancelled) — existing in v0.19
  • Shortform section grouped by platform — new under this spec
  • Folio tile (standalone scrapbook viewer entry-point) — Phase 3
  • Files tile (content tree entry-point) — Phase 3
  • Future destinations slot in as additional sections/tiles

Implications for desktop (separate feature branch)

Desktop refinement (future work, not in this feature) inherits this model. The "← Desk" back-link and menu glyph appear in equivalent desktop chrome (likely the editorial-folio header rather than the mobile masthead). The Desk's section-stack pattern becomes a multi-column or wide-grid layout where the IA can spread. The cross-cutting principle — Desk is hub, every leaf carries back + menu — holds at every breakpoint.

Why this is settled

This model was reached through five mockup passes (cross-bar 2 + four refinements). v1 enumerated 3 nav tabs; v2 demoted Manual; v3 collapsed nav to a picker; v4 dropped the picker entirely (Desk as hub); v5 refined masthead chrome. Each pass added simplification without losing capability. Reconsidering would relitigate decisions the operator has already approved.

This section is referenced by the .claude/rules/design-standards.md rule (project-wide). Surfaces that contradict it — adding a top-level tab bar, omitting the back-link on a leaf surface, omitting the menu glyph on any surface — are bugs to be fixed.

Where this lives Standards addition will go in DESIGN-STANDARDS.md's Surfaces section as a new § Studio navigation model subsection (or its own top-level § between Surfaces and Mobile vs desktop deltas). Implementation can begin once the standards update lands as a separate commit on this branch. Per the design-standards rule, the standards file is the canonical spec; mockups must comply.
