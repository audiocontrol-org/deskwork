# Design Record: ship-stage — a mediated ship waypoint that makes recording status:shipped non-optional

**Roadmap item**: `multi:feature/ship-stage`
**Date**: 2026-06-23
**Status**: pending operator approval (records as the `design-approved:` node marker)
**Scope**: ONE feature, ONE spec, ONE delivery (operator constraint, verbatim: ship "MUST be designed and implemented as ONE unit… do not split design from implementation or ship partial increments"). The phase model + the `ship` skill + the graduate-rewire + the backstop invariant + the coherence fix + the `validating` phase land together.

## problem-domain

024 (`lifecycle-compass`) and 025 (`unskippable-workflow-protocol`) made the macro-lifecycle un-skippable *up to* governing. Shipping is where it leaks:

1. **`transition:graduate` is a separate, skippable manual step.** Today `graduate` (governing→shipped) records `status: shipped` + journal + commit, but nothing *requires* it after govern converges. An agent can govern-converge, merge, release, and never fire it.
2. **PR open/merge runs through raw `gh`, off the mediated surface.** The compass never witnesses shipping happen — there is no on-rail ship step at all.
3. **The close gate and the derived phase read different sources, so they diverge.** `phase:shipped` *derives* from `record-converged impl` (the govern convergence record), so the compass reports "shipped" at **govern-converge**; the close gate (`advance --to closed`) reads the recorded `status:`, which only flips when `graduate` fires. In the graduated-but-not-status-shipped window the two **disagree** (backlog TASK-445).

**Observed live, 2026-06-23 (`multi:gap/transitive-item-closure`, v0.54.0):** after govern graduated, the manual close ran but `transition:graduate` never fired, so the roadmap `status:` stayed `in-flight` while the work was merged AND released. Caught only by luck at the close gate.

The unifying defect: **shipping is not an on-rail, witnessed, un-skippable lifecycle waypoint** — it depends on operator vigilance, which the thesis says must be made mechanical or it does not exist (*"you don't fix agents by yelling… you fix them with environmental design that makes the failure state mechanically impossible"*).

A second, adopter-facing gap surfaced during design: the **verify-before-close** discipline (*"issue closure requires verification in a formally-installed release — the agent posts evidence, the operator decides"*) is today an *implicit* operator-confirm at `advance --to closed` (031 deliberately collapsed it to a confirm with no machinery). It deserves to be a first-class, adopter-defined lifecycle phase — not a hidden confirm.

## solution-space

- **Alternative A — a mediated `ship` waypoint + a `merging` phase + a `validating` phase + recorded-status-derived `shipped` (CHOSEN).** A `/stack-control:ship` skill is the single on-rail ship step (precondition: govern-converged; open PR; merge when CI green, operator-owned) that **non-discretionarily fires `graduate`** at merge — merge and record welded so you cannot merge-without-recording. A new `merging` phase (governing → merging → shipped) names the open-PR/merge window. `phase:shipped` and every post-merge phase re-derive from **recorded `status:` + the `validated` marker** — the same sources the close gate reads — so derived==recorded by construction (the coherence half / TASK-445). A new adopter-defined `validating` phase (shipped → validating → closed) makes verify-before-close first-class. A **backstop compass invariant** refuses forward lifecycle motion (the close step + every compass-gated workflow waypoint) while a merged-but-status-in-flight item exists, catching the off-rail raw-merge residual. *Why chosen:* reuses the shipped 025 compass-enforcement pattern + the 022 gate-eval + the governed `WORKFLOW.md`; travels with `claude plugin install`; binds the adopting agent; resolves the coherence wart structurally rather than patching the close gate; and settles the whole shipped↔closed seam in one delivery (the "build as one unit" constraint).

- **Alternative B — keep `graduate` at govern-converge; add a post-merge `merged` marker + a reconcile pass (REJECTED).** Two shipped-ish states (graduated + merged) with a reconcile job to align them. This *productizes* the current divergence rather than eliminating it — exactly what TASK-445 says to remove. The reconcile pass would itself be skippable (the original defect, one layer over). Rejected.

- **Alternative C — a rule + a session-end advisory only (REJECTED as the mechanism).** "Remember to fire `graduate` after merge" as discipline, plus a session-end nag. This is the yelling the thesis rejects and what already failed: the divergence happened *with* operator vigilance available. It is also forbidden by `session-skills-never-block` if the nag became a gate. Rejected as the guarantee — but a *non-blocking* advisory surfacing at session-start/session-end is retained as a convenience layered on top of the real mechanism (never as the mechanism).

## decisions

1. **`/stack-control:ship` skill — the one on-rail ship step.** Compass precondition → confirm govern-converged (`graduate-impl` green) → open PR → merge when CI green (operator-owned) → **fire `graduate`** (record `status:shipped` + `roadmap-reconcile` + journal + commit). No skip/defer/shortcut branch (no-offroading rule).
2. **Merge and graduate are welded (enforcement layer 1).** The skill cannot perform the merge without `graduate` following — can't-merge-without-recording.
3. **`ship` is a separate operator-invoked skill, not auto-chained from `execute`.** Merge timing is operator-owned ("merge when green"). The *non-discretionary* property is the weld (merge ⇒ graduate), not auto-invocation. (Rejected alternative: `execute` auto-continues into ship — contradicts operator-owned merge timing and the two-session boundary.)
4. **New `merging` phase.** `governing → merging → shipped`. `merging` = govern-converged & `status≠shipped`; its work is `stack-control:ship`. After govern, the compass names "next: merging via /stack-control:ship," making ship a first-class waypoint.
5. **`shipped` means merged, not govern-converged.** Govern-converge becomes the *precondition* of ship, not the event that records shipped.
6. **Coherence fix (TASK-445): all post-merge phase derivation keys on recorded `status:` + the `validated` marker — the same sources the close gate reads.** `phase:shipped` re-derives from recorded `status:shipped` (was `record-converged impl`); `validating`/`closed` likewise key on recorded status + marker. Derived==recorded by construction; the divergence window closes for the on-rail path.
7. **New adopter-defined `validating` phase.** `shipped → validating → closed`. The bundled default is permissive: exit is an operator-confirm `approval-marker validated` (default behavior == 031's current confirm). The engine defines only the phase + the marker; **the adopter defines what validating *means*** (install+walk here; acceptance suite / stakeholder sign-off / nothing elsewhere) via their `WORKFLOW.md` override and their own process for setting the marker — same portable shape as `design-approved`.
8. **Backstop = a new compass invariant (enforcement layer 2).** At workflow waypoints (the close step + the compass precondition every workflow skill calls): refuse + name the dangling merged-but-status-in-flight item + direct to the reconcile command. The reconcile transition (advance the dangling item to shipped) is itself **never** blocked. (Rejected alternative: auto-reconcile — silently firing `graduate` as a side-effect of an unrelated command is surprising and would record `shipped` off a possibly-fuzzy merge-detection; refuse-and-direct matches "operator owns transitions" and the existing compass behavior.)
9. **Backstop lives at workflow waypoints, NEVER session-start/session-end** (`.claude/rules/session-skills-never-block.md`). Those skills may only *surface* the divergence as a non-blocking advisory.
10. **Honest boundary (026 pattern).** A deliberate raw `gh pr merge` outside the skill cannot be prevented; the load-bearing guarantee is the backstop gate (which detects merged-ness independently of the ship skill having run), not interception. Total prevention of a deliberate human bypass is not claimed.
11. **Built as ONE unit.** Phase model (`merging` + `validating`) + `ship` skill + graduate-rewire + backstop invariant + coherence fix land in a single spec/delivery; no partial increments.

Resolve the default signal + the portability fallback in clarify/plan.

## Amendment — 2026-06-23 (analyze F1)

`/speckit-analyze` of `specs/032-ship-stage` found that modeling BOTH a derived `phase:shipped` AND a derived `phase:validating` over the single `status: shipped` (discriminated by the `validated` marker) **deadlocks**: once validated, the derive-walk returns `shipped` while `close` fires from `validating`, leaving `closed` unreachable. Operator-confirmed resolution: **collapse to a single post-merge phase.** `shipped` is the recorded STATUS (set by `graduate` at merge); `status: shipped` derives to the `validating` phase; the `validated` marker is the `validating → closed` GATE only (not a derive discriminator). The phase chain is therefore **`governing → merging → validating → closed`** (statuses `in-flight → shipped → closed`). This preserves every approved decision above — the merge weld, "shipped means merged" (now the status), the coherence invariant, the adopter-defined validating, and the backstop — it only corrects the phase encoding. The spec/plan/data-model/contracts (032) carry the corrected model.
