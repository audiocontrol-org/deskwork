# Plan-review surface — UX audit for the comment + iterate workflow

## P0 — Same packaging defect, same effects

`http://localhost:47323/static/dist/editorial-review-client.js → 404`. Every interaction button rendered in the static HTML is dead because no client module loads:

| Affordance                          | Static HTML                  | Live behavior |
|-------------------------------------|------------------------------|---------------|
| `Edit` toggle                       | rendered top-right           | dead          |
| `Approve` / `Iterate` / `Reject`    | rendered top-right           | dead          |
| Mark-on-selection floating pencil   | rendered hidden              | never appears |
| Margin note creator (click margin)  | data-action wired            | dead          |
| Outline drawer (`Outline ↗`)        | rendered display-none        | dead          |
| Focus mode (`Focus ⛶`)              | rendered display-none        | dead          |
| Save-as-new-version (×2)            | rendered display-none        | dead          |
| Cancel-edit / Cancel-comment / Submit-comment | rendered display-none | dead    |
| `?` shortcuts modal                 | rendered display-none        | dead          |
| Auto-refresh                        | (none visible)               | n/a           |

The MARGIN NOTES column at right reads: *"Select text in the draft, then either click the floating Mark pencil above your selection — or click anywhere here in the margin to open the note."* Each instruction is a lie under the broken-JS state. Operator reads it, follows it, nothing happens, has to re-route to chat. Each round trip costs.

**Net for the comment + iterate loop today**: the studio is read-only. Comments must come through chat ("on the line that says X, change it to Y"); the agent rewrites the file; `deskwork iterate` snapshots v2; reload; repeat. That works but loses the spatial-locality benefit of inline margin notes — every comment becomes a verbal pointer instead of a visual anchor.

## P1 — Workflow gaps that persist even after the JS ships

These are real even with a working bundle:

1. **Document height is the dominant friction**. 389 markdown lines render to 20,070 px. The plan has 14 phases; navigating to Phase 7 is a long scroll. The outline drawer (`Outline ↗`) is the intended fix, but: (a) it's dead today; (b) even when working, it lives behind a button at the top right — not surfaced as a persistent navigation aid like a sticky table of contents or a phase strip across the top. For a long structured doc like a multi-phase plan, a TOC sidebar or a phase strip would beat a drawer.

2. **No diff-between-versions view.** The reviewer of v3 wants to see what changed between v2 and v3 — what comments did the agent address? The current surface shows only the current version's prose. After iteration, the operator has no in-studio way to verify "did the agent change Phase 4 like I asked?" without remembering what v2 said.

3. **No inline disposition rendering.** `deskwork iterate` accepts per-comment dispositions (addressed / deferred / wontfix). Those should land in the studio sidebar as badges. The sidebar in this version doesn't have a place to render comment cards at all — it's a single empty column.

4. **Comment list has no scaffolding.** Even if 12 comments existed, they'd render as floating margin notes attached to spans — no list view, no "next unaddressed" navigation, no count, no filter. Reviewers tracking "did I leave 5 or 6 marks?" have no way to count.

5. **State label "OPEN"** is operator-jargon. The LinkedIn surface earlier showed "IN REVIEW" as a stamp. "OPEN" is the workflow state machine's name; "IN REVIEW" is the operator's mental model. The latter is what a reviewer expects to see.

6. **No "open in $EDITOR" affordance.** The plan is `docs/source-shipped-deskwork-plan/index.md` — a real file. For substantive prose edits (rewriting a paragraph, restructuring a phase), a textarea modal is the wrong tool; the operator's actual editor is. A button "open in editor" with `code -g <path>:<line>` (or a configurable IDE protocol) would let the operator escape to their preferred environment for edits, then return to leave a margin note explaining the change.

7. **No comment-targeted iteration flow.** When the agent iterates after a review round, it reads comments from the workflow record via `deskwork review-help` and the JSON state. There's no in-studio "send these N comments to the agent" affordance. The hand-off is implicit in `deskwork iterate <slug>`. A "→ Iterate now" button (top-right) that shells out to `deskwork iterate` with the workflow id would close the loop for operators driving from the browser.

8. **Phase ordering in the document confuses the prose.** The architectural-extension section landed between Context and Phase 0, with sub-phases numbered 10–14. The studio's heading list reads: Context, Architectural extension, Phase 10, 11, 12, 13, 14, Phase 0 ... Phase 9. Phases 10–14 appear *before* Phase 0 numerically because of where the section landed. Content issue (the plan's structure), but the studio doesn't help — there's no "renumber phases" or "warn on out-of-order numbering" affordance. A doctor-rule for plan-shaped documents could surface this.

## P2 — Visual / layout

1. **Top strip and folio nav overlap visually.** The "Galley № 1 / OPEN / hint text / Edit / Approve / Iterate / Reject" strip is at y≈21–43; the masthead (`Index Dashboard Content Reviews Manual`) sits at y≈67–96. They don't quite collide on the x-axis at 1440 width but they look stacked weirdly — dashboard nav appears below action buttons. Reads as two separate horizontal bands competing for the same masthead role.

2. **Fixed margin column anchors to viewport, not prose.** As the operator scrolls down to Phase 7, the margin notes column stays glued at top-right. If a comment is anchored to a paragraph at y=12,000, the comment marker is in the prose but the comment-detail card is visible only when you scroll the prose into the viewport's right-margin band. Spatial relationship breaks for long documents.

3. **The two-row top strip on a 1440 viewport** (action row + masthead row) eats ~100 px of vertical space. On a 900-tall viewport, that's 11% of the screen real-estate before the prose even begins. For a tall doc this is precious.

4. **`?` shortcut button** at y=47 (second row, far right) is the smallest button (31×25). For first-time operators trying to discover keyboard shortcuts, it's hard to find.

5. **No favicon** — same `favicon.ico 404` as the LinkedIn surface.

## What works (so we know what to keep)

- **Newsreader serif body + Fraunces display + parchment-cream background** is the strongest part of the design. The plan reads as a *typeset* document, not a wiki page.
- **The "Galley № N" framing + state stamp** (when the stamp is re-labeled "IN REVIEW" rather than "OPEN") gives an editorial-press identity that distinguishes deskwork from generic dev tools.
- **The breadcrumb** (`DESKWORK-INTERNAL / SOURCE-SHIPPED-DESKWORK-PLAN`) in tracked-monospace is on-brand and quietly informative.
- **The MARGIN NOTES placeholder prose** is well-pitched once the underlying interactions actually work.

## Recommended sequencing

The user-facing question — "examine this surface for gaps in adding comments and editing content ahead of the next iteration loop" — is **untestable for inline comments** in the shipped state. Suggested order:

1. **First: fix the proximate packaging defect** so the JS bundles ship to adopters. Smallest possible fix is a `.gitignore` exception for `plugins/deskwork-studio/public/dist/` plus an extension to the release-workflow's bundle-verification step to cover this path. Until that lands, the comment surface is dead-on-arrival for every adopter.
2. **For the in-flight review iteration** of this plan, accept the chat-driven workaround: agent iterates based on operator comments in chat, snapshot via `deskwork iterate`, operator re-loads the URL for v2.
3. **Once JS is live**, address P1 items in priority order:
   1. Sticky TOC / phase strip for navigating long docs
   2. v(N) ↔ v(N-1) diff view
   3. Inline disposition badges from `iterate --disposition`
   4. "Iterate now" button as the in-studio handoff to `deskwork iterate`
   5. State label "OPEN" → "IN REVIEW"
4. **P2 items** are visual polish — backlog for after P1.
