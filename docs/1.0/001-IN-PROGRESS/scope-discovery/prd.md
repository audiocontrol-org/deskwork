# PRD: Scope discovery — canonize audiocontrol pilot into dw-lifecycle

## Problem Statement

The `dw-lifecycle` plugin's existing skill set (`define / setup / implement / review / ship`) carries a project through a feature lifecycle but does NOT enforce upfront scope discovery on system-wide work, and does NOT enforce sibling-enumeration on sub-agent dispatches. Both are agent-side discipline gaps that have been measurably expensive in real work — the audiocontrol pilot's motivation: *"the Roland S-330/S-550 v3 redesign (May 2026), which spent ~230 operator turns over 60 hours doing brute-force discovery the agent should have done in 10–15 minutes at session start. ... The protocol treats agent-side enforcement as code, not directives — passive rules in CLAUDE.md and agent prompts have demonstrably failed against persistent pathologies in this repo. Every gate the protocol introduces is code-shaped: it rejects the bad shape mechanically, not by asking the agent to remember."* Audiocontrol built and validated the protocol in-repo: `/scope-inventory <slug>` upfront discovery, `/scope-widen "<complaint>"` mid-implementation widening, pre-commit clone-detector gate, sub-agent dispatch wrapper with `Searched/Included/Excluded` grammar enforcement, Step 0 refactor-preconditions, anti-patterns + adopter-manifests + editor-symmetry + deprecation-queue scan types. Paper-tested at 87.5% coverage against the s550 redesign's 32 documented surfaces. This feature canonizes the protocol into `dw-lifecycle` so any project gets it. The audiocontrol pilot's repo-local copy becomes one adopter among many; the deskwork repo becomes the second adopter (the canary), exercising the canonization end-to-end against the in-flight `graphical-entries` feature.

**Phase 11 extension (2026-05-26).**

**Thesis: Phase 11 is behind-the-scenes code hygiene, not an operator-driven workflow.** The orchestrator-agent (not the operator) drives the discovery loop, the LLM-as-judge, the external auditor, the catalog edits, the controller's cadence adjustments, and the wrong-decision recovery. The bulk of Phase 11's machinery runs autonomously inside the `/dw-lifecycle:implement` cycle on every turn; the operator never sees it unless the orchestrator can't decide. The operator's role narrows to setting goals (via the workplan) and resolving the rare escalations the orchestrator surfaces when policy + skills + metrics don't yield confident next action. The operator NEVER triages findings at the line level; the agent mediates everything to architecture-scale dispositions when human input is actually needed. The architectural failure mode this prevents — codified across multiple operator corrections during the design conversation — is human-discipline-dependence: any system that relies on the operator to remember to invoke a sub-step, edit a YAML, or audit a finding fails because human discipline doesn't scale and isn't durable. Phase 11 mechanizes the discipline.

The trigger for Phase 11 is the first dogfood-cycle finding ([#315](https://github.com/audiocontrol-org/deskwork/issues/315)) which surfaced a structural gap: the discovery agents act as **inventory** (match against a fixed hardcoded vocabulary) rather than **discovery** (surface novel anti-patterns the catalog doesn't yet know about). Canonical repro from the audiocontrol `feature/akai-harmonization` branch: a component with ZERO canonical-primitive consumers + ≥14 utility-class hits passed every scanner and every audit; was caught only when the operator inspected a screenshot. The general failure mode is operator-trust: a green "discovery" report is read as evidence of no novel anti-patterns, when it's really evidence of no already-registered matches. The catalog ages out as the codebase evolves and new shapes appear; the tooling has no mechanism to surface those novel shapes for triage. Phase 11 closes the gap by transforming the surface from one-shot inventory into a continuous self-correcting discovery loop — embedded in `/dw-lifecycle:implement`, running autonomously, surfacing to the operator only on escalation.

**Phase 24 extension (2026-06-03).** Three open GitHub issues filed 2026-06-03 by an agent driving `feature/deskwork-plugin` ([#401](https://github.com/audiocontrol-org/deskwork/issues/401), [#402](https://github.com/audiocontrol-org/deskwork/issues/402), [#403](https://github.com/audiocontrol-org/deskwork/issues/403)) surface a structural mistake in the audit-finding lifecycle architecture: enforcement was wired into git commit-msg + pre-push hooks. The pathologies are concrete — a 1-commit task ballooned into 5 commits + 3 rounds of barrage findings ([#401](https://github.com/audiocontrol-org/deskwork/issues/401)); a trailing bookkeeping commit produces a coverage-gate ratchet with no terminal state ([#402](https://github.com/audiocontrol-org/deskwork/issues/402)); the aggregate bookkeeping-to-substance ratio is ~3:1 on a small task ([#403](https://github.com/audiocontrol-org/deskwork/issues/403)). Yesterday's v0.35.0 release required three `--no-verify` pushes for bookkeeping commits the gates refused. But the root failure is architectural, not procedural: the audit-finding gates (`check-implement-hook-ran`, `check-implement-hook-coverage`) are NOT installable by adopters — they exist only in this repo's hand-rolled `.husky/`. The operator's framing, verbatim: *"we've made a core architectural mistake by wiring enforcement into git. We can't expect adopters to install our git hooks, so we shouldn't rely on them here — it distorts our perception of what's working and not. Instead, we need to figure out how to bake enforcement into what's actually available in the plugin. For example, we can put gates into session-start, session-end and implement, but not into git hooks."* The bookkeeping pathology is OUR pathology — adopters wouldn't experience it because they don't have the gates that produce it. Conversely, every gate UX call we've made (TDD task shape, `Closes AUDIT-X` trailer, `apply-audit-flips` workflow, dampener, hook-run-log) has been judged through our own invocation via `npx tsx plugins/...` — none of it has been driven through a real install. The structural pre-commit chain (`check-clones`, `check-anti-patterns`, `check-adopters`, `check-disposition-survivor`, `check-editor-symmetry`) IS plugin-installable via `install-scope-discovery-hooks`, but the install requires an adopter to know about husky and run the install verb separately. Same architectural problem, smaller volume. Phase 24 closes the gap: zero git-hook reliance, full discipline composed into skill bodies + CLI verbs adopters get by installing the plugin. The principle generalizes the existing rule *"Use the deskwork plugin only through the publicly-advertised distribution channel"* to enforcement specifically.

**Phase 25 trigger (2026-06-03).** During Phase 24 scoping, the editor-symmetry terminology surfaced as a leaked-domain-term pathology: the audiocontrol pilot named the parallel-top-level-module concept `editor` because its modules ARE editors (S-330 editor, S-550 editor — Roland-sampler editors, each under `modules/<slug>-editor/`). The dw-lifecycle canonization preserved the term verbatim because renaming would have invalidated the Phase 3 schema. Every non-audiocontrol adopter mentally translates `editor` → `module`. The irony: this is exactly the leaked-domain-terminology pathology that scope-discovery exists to surface. Phase 25 pays the schema-stability cost (separate from Phase 24 because mechanical rename should not mix with architectural reframe in commit history).

**Phase 26 trigger (2026-06-03).** Mid-session, between Phase 24's gate dismantling and the `/dwi` handoff, the workplan's bloat surfaced as a `/dwi`-efficiency blocker: 4477 lines accumulated across 25 phases, most of them completed work. The active workplan was buried under historical fix-task blocks; `/dwi`'s next-unchecked-task walker landed on stale fix-tasks instead of the active Phase 24/25 work. A manual archive operation moved 16 completed phases (1-5, 9-10, 13-14, 16-19, 21-23) to a sibling `workplan-archive.md` and reduced the active workplan to 1036 lines (77% smaller). The manual operation revealed the pattern as generalizable: long-running features accumulate bloat; the audiocontrol pilot has the same problem. Phase 26 productizes the manual operation as a CLI verb (`dw-lifecycle archive-phases` + `unarchive-phases`) and teaches the auto-positioner to honor the workplan-archive-ledger annotation so promote-findings doesn't collide with archived fix-task IDs.

## Solution

Move the audiocontrol-piloted Scope Discovery Protocol's CODE (scanners, validators, discovery agents, dispatch wrapper, Step 0 enforcement) into the `dw-lifecycle` plugin as plugin-shipped TypeScript modules, while leaving the CONFIG (dispositioned baselines + anti-patterns + adopter-manifests + editor-symmetry + deprecation-queue) project-owned at `<projectRoot>/.dw-lifecycle/scope-discovery/`. Add ~18 new `/dw-lifecycle:*` slash commands, update 5 existing skills (`define / implement / review / doctor / customize`) for auto-invocation at the right pipeline phase with per-phase opt-out flags. Pre-commit hook + dispatch wrapper + agent-prompt mirrors land as opt-in scaffolds via dedicated install skills. The deskwork repo itself becomes the second adopter (the canary, with audiocontrol's pilot the first); the in-flight `graphical-entries` feature provides the real-world validation case — its `/dw-lifecycle:setup` auto-runs `/scope-inventory` and the resulting paper-test coverage matrix is the v1 acceptance signal.

**Phase 11 extension (2026-05-26).** Per the thesis: this is behind-the-scenes code hygiene driven by `/dw-lifecycle:implement`, not an operator workflow. The additions below describe the AUTONOMOUS machinery the orchestrator runs every turn:

- **Pattern-type vocabulary widening.** Beyond today's positive-match regex: negative-space (files matching glob G that DON'T contain shape S), coverage-metric (`glob × shape → ratio`), statistical-outlier (anomaly detection across sibling components), unmatched-shape clustering (synthesis-layer; ranks novel shapes for triage), and semantic (LLM-augmented). Polymorphic type-handler dispatcher; per-project YAML override extends the catalog with any type.
- **Status-marked catalog entries.** Every catalog entry (anti-patterns, adopter-manifests, editor-symmetry, deprecations, pattern-matrix, clones) gains a `status:` field — `pending / blessed / cursed / ignore / tracked-holdout / withdrawn` — plus a `provenance:` block linking the edit to its origin (operator-authored / orchestrator-agent / llm-judge-proposed / install-seed / promoted-from-candidate, with timestamps + audit-log cross-references). Discovered candidates accumulate in the SAME catalogs as registered entries (no separate ledger).
- **Orchestrator-agent mediation.** Operator dispositions at architecture-scale ("class C7 of shapes is a design-system holdout"); orchestrator-agent translates to line-level catalog edits (append novelty vs refine existing entry; auto-derive regex/glob/shape from candidate evidence; populate excludes_paths). The operator never edits regex; the agent never picks the architectural disposition.
- **Self-correcting controller.** Measures codebase-state metrics (classification completeness, coverage, violation density, surface uniformity, catalog stability, discovered-candidate rate, disposition latency) + their derivatives (drift rate, correction rate) + auditor-correction-rate (the truth signal). Adjusts the Loop's cadence + intensity based on measurement, not doctrine. Cold-start defaults to max intensity, ratchets down as the controller earns confidence.
- **`/dw-lifecycle:implement` augmentation.** Per-turn audit/judge stack runs autonomously: read audit log → internal LLM-judge pass → take action → fire external LLM auditor. Hook-cadence is workplan-dependent + controller-tuned. Termination criteria negotiated per-cycle. Resumability state durable in catalogs + audit log.
- **Multi-source LLM ensemble.** Internal LLM-judge (in-band, every turn) + external LLM auditor (out-of-band, fired by orchestrator every turn, results materialize in audit log for next turn) + audit-log (durable memory). Judge and auditor are independent (different model/prompt scaffolds; auditor cannot self-grade the judge). Human is escalation-only target — rare, high-information, asynchronous-friendly.
- **Wrong-decision recovery primitives.** Reversible dispositions via `withdrawn-<finding-id>` status (mirrors existing audit-log convention; catalog entries never deleted). Trust calibration updates per auditor-correction event. Systematic-wrongness response: a CLASS of decisions consistently wrong routes to escalation by default until evidence improves.
- **Cross-surface application.** The new primitives (status + provenance + polymorphic type dispatch) apply uniformly across ALL registry-driven scanners: anti-patterns, adopter-manifests, editor-symmetry, deprecations, pattern-matrix, and the regime-holdout-detector that fuses them.
- **Tooling-feedback closure → audit-log import workflow.** Closure-marked TF entries auto-promote to scope-discovery audit-log AUDIT-`<date>`-`<NN>` entries with cross-reference; doctor rule surfaces stale TF entries; the loop closes.

Phase 11 captures exhaustively per the capture-mode rule. Scoping — which sub-tasks ship in which release — is operator-driven.

**Phase 12 extension (2026-05-29).** Triggered by the canary [#349](https://github.com/audiocontrol-org/deskwork/issues/349) §2 framing of the manually-run codex audit as an operator-attention cost the protocol can't absorb at scale. The audit posture today has three layers: in-band self-audit (orchestrator-loop, same model + same context — blind to its own failure modes), the SDD two-reviewer cycle (spec + quality sub-agent dispatches — catches real BLOCKING bugs per canary §1a), and the manually-run codex audit (operator pastes work into a separate Codex session — demonstrably finds what Claude misses but requires manual invocation + paste + finding-by-finding triage). The third layer's audit quality is currently a function of the operator's intermittent capacity to run the audit by hand; manual discipline doesn't scale and isn't durable. Phase 12 ships a third automated audit surface — a multi-model audit barrage — that fires installed CLI tools (`claude`, `codex`, `gemini`) in parallel against a uniform prompt, persists raw per-model output, and surfaces a structured triage step. **CLI-based, not API-based** — usage-based pricing means the barrage can fire freely without per-call cost arithmetic; per-CLI auth is already configured in the operator's environment; subprocess orchestration is a well-trodden plugin pattern (already in use for `gh` / `git` / `npx tsx` / `jscpd`). The phase is ADDITIVE — not a replacement for the in-band self-audit or the SDD review cycle; all three audit surfaces stay active. This phase implements Design A from `ROADMAP.md` § "Audit-barrage feature shape". Design B (auto-fire at lifecycle waypoints + meta-audit synthesizer) and Design C (continuous background daemon) are deferred to follow-up cycles after Design A is stable + the operator has accumulated cross-model finding patterns. Workplan § Phase 12 captures all 7 sub-tasks exhaustively per the capture-mode rule.

Phase 12 captures exhaustively. Scoping is operator-driven; the agent does not pre-cut.

**Phase 13 extension (2026-05-29).** Triggered by operator-side observation during the Phase 12 self-dogfood — the agent that ran the audit-barrage lifted 11 findings into the canonical audit-log, then went straight from "findings lifted" to "fix dispatch" without scoping the fixes into the workplan. Findings then got addressed ad-hoc in a parallel session. Both directions (skip-the-workplan-and-just-fix; skip-the-workplan-and-defer-to-GH-issue) are the same failure mode: the workplan stops representing the source of truth for what the implementation loop is supposed to pick up. The operator's framing, verbatim: *"Filing a bug report isn't good enough. It MUST BE SCOPED INTO THE WORKPLAN, otherwise it won't get picked up by the implementation loop. Unless there's truly a good reason NOT to fix a problem, it should be relentlessly scoped into the workplan, not relentlessly deferred — ESPECIALLY problems with the implementation underway. A broken implementation is not done — it's broken. And, along with the discipline to scope the fix, TDD principles should apply such that a test that exercises the bug is written before the fix is implemented — and the implementation isn't considered a candidate for completion until tests are green."*

Phase 13 ships the structural tooling that makes this discipline mechanical:

- **NEW `/dw-lifecycle:promote-findings` skill + CLI verb.** Walks `audit-log.md` for `Status: open` entries; **default disposition is "scope into the workplan as a TDD-first fix task"**. The agent CANNOT pick deferral. Only the operator can — and only with a substantive-reason validator pass (≥40 chars; gaming-phrase reject list per hygiene's `promote-deferrals`).
- **TDD-first workplan task shape** — every promoted finding renders into a workplan task block whose Steps mandate: (1) write a failing test exercising the bug, (2) confirm it fails, (3) implement the fix, (4) confirm it passes, (5) commit with `Closes AUDIT-<id>` in subject. Operator confirms placement (which phase, where in the phase); task content is not negotiable.
- **`/dw-lifecycle:implement` strict refusal gate** — refuses to advance to the next task while ANY finding on the current feature has `Status: open`. No `--ignore-open-findings` escape hatch in v1; per operator decision, err on rigidity. (If proves unworkable: revisit relaxation options.)
- **Mechanical TDD enforcement** — a doctor rule + commit-msg gate verify that fix-finding tasks marked `[x]` actually have a passing test file at the path cited in Step 1. Not a markdown checkbox the agent can flip; a mechanical check the project's existing gate machinery enforces.
- **Closure-side automation** — `Closes AUDIT-<id>` commits auto-flip audit-log entries to `Status: fixed-<sha>`; release process flips `fixed-<sha>` → `verified-<date>` for findings in the release range; a NEW `re-audit-fixed-findings` skill verifies findings no longer surface post-release.

Phase 13 is the operator-discipline-displacement counterpart to Phase 12. Phase 12 produces the findings; Phase 13 forces them through the workplan into completion. The two ship together; Phase 12 without Phase 13 would just produce more findings to ad-hoc rationalize away.

Phase 13 captures exhaustively. Scoping is operator-driven; the agent does not pre-cut.

**Phase 24 extension (2026-06-03).** The architectural principle: enforcement lives in surfaces an adopter installs and runs (skills + CLI verbs), not in git hooks. A discipline that can only fire from `.husky/` does not exist for an adopter who follows the public install path. Phase 24 delivers a single, atomic reframe with three movements:

- **Demolition.** Delete `.husky/commit-msg`; remove the audit-gate block from `.husky/pre-push`; gut the structural-chain block from `.husky/pre-commit`. Retire `check-implement-hook-ran` + `check-implement-hook-coverage` subcommands + skills + tests + sources. Retire `--upstream-base-ref` flag + per-SHA `hook-run-log.jsonl` write logic + `last-hook-run.json` marker + boot-case guards. Phases 21 / 22 / 23 retire as vestigial; Phase 17's "with teeth" portion retires; Phase 15's "gate" half retires. The install machinery (`install-scope-discovery-hooks` + `uninstall-scope-discovery-hooks` + `hooks-installed.json`) retires entirely.
- **Relocation.** Structural chain (`check-clones`, `check-anti-patterns`, `check-adopters`, `check-editor-symmetry`) composes into `/dw-lifecycle:session-start` (advisory snapshot) and `/dw-lifecycle:implement` end-of-task (enforce). `/dw-lifecycle:implement` end-of-task composes the **full Phase 15 audit-finding chain** (per AUDIT-20260603-28 correction): `audit-barrage --output-run-dir` → `audit-barrage-lift --apply` → `promote-findings --auto` → `check-open-findings` (refuse-to-advance gate) → `apply-audit-flips` (close already-fixed) → `check-fix-task-tdd` (advisory). The lift/promote/check-open-findings chain is the Phase 15 machinery that turns raw barrage output into workplan-tracked open findings; without it the HIGH-finding refusal gate has no mechanism. Phase 24 preserves the Phase 13 + 15 finding lifecycle UNCHANGED in concept; only the firing location moves from `.husky/commit-msg` to the skill body. `check-disposition-survivor` + no-bare-TBDs + no-open-findings-without-disposition compose into `/dw-lifecycle:session-end`. `check-refactor-preconditions` becomes Step 0 of `/dw-lifecycle:review`; the structural chain runs there too as a PR-readiness gate; `check-editor-symmetry` runs as a fleet snapshot. The Phase 20 Task 2 retirement of `/dw-lifecycle:review` reverses: `review` becomes the *primary* enforcement surface, not deprecated.
- **Migration.** A one-shot `dw-lifecycle uninstall-everything-hook-related` verb removes managed blocks from `.husky/`, deletes `hooks-installed.json`, surfaces a report. Release notes call this a breaking change. Plugin README + new MIGRATING.md capture the upgrade path.

Phase 24 is the operator-discipline-displacement counterpart to Phase 13. Phase 13 forced findings through the workplan into completion; Phase 24 ensures the discipline runs through surfaces adopters install. Phase 13's audit-finding lifecycle ships unchanged in CONCEPT; what changes is where the gates *fire from*. Without Phase 24, the discipline is a UX we don't ship.

The phase closes by running a real implementation task on this branch using the new shape (no `.husky/` enforcement active); measuring bookkeeping touches per finding (target <2:1, down from ~3:1); confirming zero `--no-verify` or `git reset` invocations needed; confirming the structural chain still catches the regressions it caught when wired as a hook; confirming `/dw-lifecycle:implement` end-of-task audit-barrage produces equivalent finding coverage to the retired `check-implement-hook-ran` gate.

**Phase 25 extension (2026-06-03).** Phase 25 pays the schema-stability cost: rename `editor` → `module` everywhere in the scope-discovery layer. **Phase 26 extension (2026-06-03).** Phase 26 ships two CLI verbs + one auto-positioner enhancement: (1) `dw-lifecycle archive-phases --feature <slug> --phases <range>` moves completed phase sections from `workplan.md` to `workplan-archive.md`, updates the workplan-archive-ledger comment with the new archived ranges + `next-fix-task-id`. (2) `dw-lifecycle unarchive-phases` is the symmetric reversal verb. (3) `promote-findings`'s auto-positioner reads the ledger annotation FIRST and uses `max(ledger.next-fix-task-id, scan-of-workplan-max + 1)` so archive operations don't break ID continuity. The verbs refuse archiving phases with ANY unchecked task BY DEFAULT, with an explicit `--allow-vestigial <reason>` escape (≥40-char substantive-reason requirement) for retired-vestigial phases. The default refusal catches accidental over-archive; the escape mechanizes the case the manual 2026-06-03 archive operation needed for Phases 17/22/23 — phases vestigial under Phase 24's retirement decision whose unchecked steps are not actionable (per AUDIT-20260603-37 + the workplan-archive header's "completed work OR vestigial per a later phase's retirement decision" framing). The ledger records the operator-supplied reason next to the phase entry so future readers see WHY an incomplete phase was archived. A doctor rule `workplan-archive-ledger-coherence` validates that the ledger matches the archive file's actual content. `/dw-lifecycle:complete` optionally runs `archive-phases --all` at feature-completion time so the `003-COMPLETE/` archive isn't a 4000-line workplan. The audiocontrol pilot gets the same benefit on its own bloated workplan.

Phase 26 is independent of Phase 24 + Phase 25 — can land in any order. The manual archive done 2026-06-03 already works (the ledger captures `next-fix-task-id: 5.124`); Phase 26 mechanizes the second-and-onward use.

**Phase 25 MUST ship after Phase 24** (correction per AUDIT-20260603-24): Phase 24's Relocation movement adds new `check-editor-symmetry` call sites in `/dw-lifecycle:session-start` and `/dw-lifecycle:review` skill bodies. Phase 25 Task 1's inventory enumerates the rename surface; if drafted before Phase 24 lands, the inventory would undercount the live call sites. ## Technical Approach

scope-discovery ships in 11 phases. Plugin-side CODE phases land first (Phases 1–7), then install / migration / doctor wiring (Phases 8–9), then the deskwork canary install + `graphical-entries` paper-test deliverable (Phase 10 — the v1 ship). Phase 11 adds the continuous self-correcting discovery loop architecture surfaced by the first dogfood-cycle finding. The full design rationale, schema specifications, dispatch wrapper enforcement rules, real-world validation case mapping, and risk analysis live in the design spec at `docs/superpowers/specs/2026-05-24-scope-discovery-design.md`. Plugin-default + project-override boundary per THESIS Consequence 3: CODE in plugin; CONFIG in project; per-file scanner overrides via `/dw-lifecycle:customize scope-discovery <name>`. Integration with the existing dw-lifecycle pipeline is hybrid: explicit slash commands + auto-invocation in `define/implement/review` with per-phase opt-out flags.
