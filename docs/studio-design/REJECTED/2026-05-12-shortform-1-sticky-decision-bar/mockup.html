What I learned shipping mobile-first.

Three iterations on the state-stamp pattern; what finally worked was retiring it on mobile.

Press-check vocabulary, take three. I keep going back to this question: what should a reviewed draft actually look like on screen?

The first iterations leaned on rotated rubber stamps — they read on desktop where stamps spread horizontally, but on mobile they cluster and fight the row's natural composition.

Letterpress tags were no better — a label on a rectangle is a label on a rectangle no matter how you draw it.

The third draft retires state surfacing entirely on mobile. Where it matters, stage hierarchy and the action affordance itself communicate state. Where it doesn't, absence is information too.

I'm calling it: stamps stay on desktop, mobile rows render clean.

Why I stopped writing CSS by hand

I keep two flavors of CSS in my head these days. One is the rules of the game — flexbox alignment, what min-width: 0 does to grid items, the cascade. The other is the syntax — every comma, every semicolon, every :hover selector I have to type a thousand times.

The first flavor matters. It's the part where you make a decision: this header is sticky, this column shrinks before that one, the gap between siblings is var(--space-3). The second flavor is just keystroke tax.

Tailwind's pitch was the second flavor — let me write flex items-center gap-3 instead of authoring a stylesheet. Fine. But what I want now is closer to the first: tell me my decisions are wrong, not retype them for me.

The shape that's worked, on this project specifically: token vars in one place, semantic selectors that read like prose (.er-strip-galley not .flex.items-center.gap-2), media queries scoped to a single data-attribute.

I can grep my CSS the way I grep code. The decisions are visible.

Press-check vocabulary helped. Naming colors after presscheck objects — paper, ink, red-pencil, proof-blue — meant the next decision wasn't what hue, it was what role. That cuts time.
