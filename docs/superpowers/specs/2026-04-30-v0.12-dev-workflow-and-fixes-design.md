# v0.12.0 — Local dev workflow + critical post-v0.11.1 fixes

**Date:** 2026-04-30
**Status:** approved
**Branch:** feature/deskwork-plugin
**Target release:** v0.12.0 (major)

## Context

v0.11.1 shipped the entry-centric Phase 30 redesign. Dogfood against this project's own calendar (2026-04-30 session) surfaced a cluster of bugs filed as [#137](https://github.com/audiocontrol-org/deskwork/issues/137)–[#146](https://github.com/audiocontrol-org/deskwork/issues/146). The most critical:

- **Migration data loss.** Phase 30's `migrateCalendar` repair class generates per-entry sidecars with artifact paths derived from a slug heuristic — `<contentDir>/<slug>/index.md` for Drafting, `<contentDir>/<slug>/scrapbook/idea.md` for Ideas — and ignores the actual `sourceFile` recorded in the legacy ingest journal. Projects with custom doc layouts (`docs/1.0/...` here) get phantom `file-presence` failures from `deskwork doctor`. The migration also drops review-workflow state on the floor instead of mapping iteration counts and approval state from the legacy workflow records onto the new sidecar's `reviewState` / `iterationByStage`. ([#140](https://github.com/audiocontrol-org/deskwork/issues/140), [#141](https://github.com/audiocontrol-org/deskwork/issues/141))
- **Studio half-migrated.** The per-entry review URL routes by entry UUID (Phase 30 Task 35) but the page renderer is still the legacy workflow-object code path. It displays state strings (`applied`) that aren't valid Phase 30 `ReviewState` values (`'in-review' | 'iterating' | 'approved'`), and exposes no UI for the new entry-stage universal verbs (`approve` / `block` / `cancel` / `induct` / `publish`). The dashboard's empty-Planned hint explicitly tells the operator to drop to CLI: *"Nothing planned. /deskwork:approve <slug> to graduate an idea."* ([#146](https://github.com/audiocontrol-org/deskwork/issues/146))
- **Adopter-facing surface drift.** `repair-install.sh` enumerates plugin versions from `$PATH` (re-restoring orphan caches when PATH is stale within a session) and walks every registry entry without filtering by `scope`/`projectPath` (cross-project leak — restored writingcontrol's project-scope cache into this session's deskwork-plugin worktree). `deskwork --help` advertises retired verbs (`plan`, `outline`, `draft`, `pause`, `resume`, `review-*`) as if usable and omits the new universal verbs. ([#137](https://github.com/audiocontrol-org/deskwork/issues/137), [#138](https://github.com/audiocontrol-org/deskwork/issues/138), [#139](https://github.com/audiocontrol-org/deskwork/issues/139))

In addition, the project lacks a working local dev workflow. Every code change to the plugin currently requires a publish-bump-tag-marketplace-update-cache-restore round trip. The bin shim has workspace-symlink detection (`path 1` in `bin/deskwork`'s resolution order), but workspace symlinks aren't currently in place, no package has a `dev` script, and the studio's client assets are bundled by in-process esbuild on startup with no watch mode.

## Goals

1. **Standard local dev workflow.** Edit code in `packages/studio/src/*` (or `public/src/*`), see changes in the browser without publishing. Edit code in `packages/cli/src/*` or `packages/core/src/*`, run from source via `claude --plugin-dir plugins/deskwork`. The bin shim's workspace-symlink detection becomes the dispatch path during development.
2. **Fix the highest-leverage v0.11.1 issues.** Tier A — migration data loss ([#140](https://github.com/audiocontrol-org/deskwork/issues/140), [#141](https://github.com/audiocontrol-org/deskwork/issues/141)) and the studio's per-entry review surface ([#146](https://github.com/audiocontrol-org/deskwork/issues/146)) — these break the dogfood loop on this project's own calendar. Tier B — CLI `--help` ([#139](https://github.com/audiocontrol-org/deskwork/issues/139)), `repair-install.sh` ([#137](https://github.com/audiocontrol-org/deskwork/issues/137), [#138](https://github.com/audiocontrol-org/deskwork/issues/138)) — high-value, low-effort polish.

## Non-goals

- **Hot module replacement for the CLI/core packages.** `tsx --watch` for the studio server is in scope; full HMR for non-web code paths is not. Manual rebuild + rerun via `claude --plugin-dir plugins/deskwork` is fine for CLI/core during development.
- **Tier C issues** — [#142](https://github.com/audiocontrol-org/deskwork/issues/142) (pipeline-stage design question), [#143](https://github.com/audiocontrol-org/deskwork/issues/143)/[#144](https://github.com/audiocontrol-org/deskwork/issues/144)/[#145](https://github.com/audiocontrol-org/deskwork/issues/145) (studio routing 404s), [#109](https://github.com/audiocontrol-org/deskwork/issues/109)/[#110](https://github.com/audiocontrol-org/deskwork/issues/110)/[#112](https://github.com/audiocontrol-org/deskwork/issues/112) (older studio polish). These get a "next sprint" note in the workplan, not a fix in v0.12.0.
- **CI test infrastructure.** Per project rule "no test infrastructure in CI." Smoke and unit tests stay local-only.
- **Backwards compatibility for entry sidecar schema.** We're our own only adopter; [#141](https://github.com/audiocontrol-org/deskwork/issues/141)'s fix may add fields to the sidecar schema and re-running `deskwork doctor --fix=all` is the migration path.

## Section 1 — Local dev workflow (Vite middleware in Hono)

### Approach

Mount Vite as middleware inside the Hono server, gated on `DESKWORK_DEV=1`. Production path (in-process esbuild → `.runtime-cache/dist/`) is untouched. Vite is a devDependency only.

### Components

1. **Workspace symlinks via `npm install` at root.** The plugin shells (`plugins/deskwork`, `plugins/deskwork-studio`, `plugins/dw-lifecycle`) already declare workspace package deps in their `package.json` (e.g. `"@deskwork/studio": "0.11.1"`). `npm install` at the workspace root creates `plugins/<plugin>/node_modules/@deskwork/<pkg>` as symlinks into `packages/<pkg>/`. The bin shim's path-1 detection (`<plugin-root>/node_modules/@deskwork/<pkg>` is a symlink) takes over with no install step.
2. **`dev` script on `@deskwork/studio`.**
   ```json
   "dev": "DESKWORK_DEV=1 tsx --watch src/server.ts"
   ```
   `tsx` runs the server directly from TS source; `--watch` restarts the process when any imported file changes; the env var is the dev-mode flag the server reads.
3. **`packages/studio/src/server.ts` branches on `DESKWORK_DEV`.** In dev mode, skip the in-process esbuild step and create a Vite dev server in middleware mode (`createServer({ server: { middlewareMode: true }, root: clientSrcDir, appType: 'custom' })`). Mount `vite.middlewares` on the request pipeline so it intercepts `/@vite/`, `/src/`, and any path under the client source root; non-client requests fall through to Hono. In prod, the existing path runs untouched.
4. **Page-rendering helper for client `<script>` tags.** New `packages/studio/src/lib/client-script.ts` exports `clientScriptTag(name: string): string`. In dev: emits `<script type="module" src="/src/${name}.ts"></script>`. In prod: emits `<script type="module" src="/static/dist/${name}.js"></script>`. All page renderers (`dashboard.ts`, `content.ts`, `editorial-help.ts`, `editorial-review.ts`, `editorial-review-shortform.ts`, `index.ts`) use this helper instead of inlining script tags.
5. **`packages/studio/vite.config.ts`** (new, minimal). Vanilla TS, no plugins (the studio's client code doesn't use a framework).
6. **`vite ^5` as a `devDependency`** of `@deskwork/studio`.
7. **Doc** in `packages/studio/README.md` — three-line quickstart: `npm install` → `npm run dev --workspace @deskwork/studio` → `claude --plugin-dir plugins/deskwork-studio`.

### Files touched (Section 1)

| File | Change |
|---|---|
| `packages/studio/package.json` | + `dev` script, + `vite` devDep |
| `packages/studio/src/server.ts` | branch on `DESKWORK_DEV`, mount Vite middleware in dev |

## Section 2 — Fix scope

### Tier A — must-include

#### [#140](https://github.com/audiocontrol-org/deskwork/issues/140) — Migration derives sidecar paths from heuristic

**Problem.** `migrateCalendar` computes the expected artifact path from `<contentDir>/<slug>/index.md` (Drafting) or `<contentDir>/<slug>/scrapbook/idea.md` (Ideas). Projects whose markdown isn't laid out at those paths get phantom `file-presence` failures.

**Fix.** Add `artifactPath` to the entry sidecar schema (relative to `contentDir`). Migration reads `sourceFile` from the legacy ingest journal entries (`.deskwork/review-journal/ingest/*.json`) and writes that as `artifactPath` on the sidecar. The doctor's `file-presence` validator and the studio's content/review resolvers consume `artifactPath` directly. New entries (created via `add` / `outline` / `induct`) populate `artifactPath` at creation time using the existing slug→path heuristic, which is correct for new entries created in the slug-shaped layout — only legacy migrations break it.

**Files.** `packages/core/src/schema/entry.ts`, `packages/core/src/doctor/migrate.ts`, `packages/core/src/doctor/validate/file-presence.ts`, `packages/studio/src/lib/entry-resolver.ts`.

#### [#141](https://github.com/audiocontrol-org/deskwork/issues/141) — Migration doesn't carry workflow state

**Problem.** `migrateCalendar` initializes sidecars with `iterationByStage: {}` and no `reviewState`, even when legacy `review-journal/pipeline/*.json` records have iteration history and `state: "applied"`.

**Fix.** During migration, walk `.deskwork/review-journal/pipeline/*.json`, group by `entryId` (or by `slug` for legacy records that pre-date `entryId`), and populate the sidecar's `iterationByStage[currentStage]` with the highest `currentVersion` per group. Translate workflow `state` to entry `reviewState`:
- legacy `"applied"` → `"approved"`
- legacy `"open"` / `"iterating"` → `"in-review"` / `"iterating"`
- legacy `"reverted"` / `"cancelled"` → no `reviewState` (drop)

**Files.** `packages/core/src/doctor/migrate.ts`, vitest cases.

#### [#146](https://github.com/audiocontrol-org/deskwork/issues/146) — Studio per-entry review surface uses legacy renderer

**Problem.** `/dev/editorial-review/<entry-uuid>` renders from the legacy pipeline-workflow record. Shows invalid Phase 30 state strings. No UI for entry-stage transitions.

**Fix.** Rewrite the per-entry review page renderer to consume the entry sidecar:
- Render `currentStage`, `iterationByStage[currentStage]`, `reviewState` (the three valid Phase 30 values).
- Replace the legacy workflow-action controls with entry-stage actions: visible buttons for `Approve` (graduate to next stage), `Block`, `Cancel`. Keyboard shortcuts can stay for parity but bind to entry actions, not workflow actions.
- Mutation endpoints: `POST /api/dev/editorial-review/entry/<uuid>/approve` (and `/block`, `/cancel`). Server-side calls `@deskwork/core`'s entry-stage transition helpers (`iterateEntry` already exists; verify and add sibling helpers as needed).
- Workflow-level review (margin notes, `Edit` mode for content) stays — that's per-entry-version content review and is orthogonal to entry-stage transitions.

**Files.** `packages/studio/src/pages/entry-review.ts` (rewrite renderer to consume entry sidecar instead of the legacy workflow record), studio routes file (locate during implementation — likely co-located with `entry-review.ts` or under `packages/studio/src/routes/` once that directory is confirmed), `packages/core/src/iterate/iterate.ts` (existing; add sibling `approveEntryStage` / `blockEntry` / `cancelEntry` helpers alongside the existing `iterateEntry`).

### Tier B — high-leverage, small

#### [#139](https://github.com/audiocontrol-org/deskwork/issues/139) — CLI `--help` out of sync

**Fix.** Rewrite the help renderer along Phase 30's verb structure:
- Drop `plan`, `outline`, `draft`, `pause`, `resume`, `review-start`, `review-cancel`, `review-help`, `review-report` from the listing (the runtime gate carries the migration message).
- Add `block`, `cancel`, `induct`, `status` under "Lifecycle" or "Universal verbs."
- Re-categorize `iterate`, `approve`, `publish` as universal verbs, not "review loop."

**Files.** the CLI help renderer (one file).

#### [#137](https://github.com/audiocontrol-org/deskwork/issues/137) — `repair-install.sh` re-restores orphan caches when PATH is stale

**Fix.** Drop the PATH source from `versions_referenced()` (lines 122–135 in `scripts/repair-install.sh`). Sources 2 + 3 (registry + canonical marketplace manifest) cover legitimate versions. PATH-derived entries are never authoritative.

**Files.** `scripts/repair-install.sh` (one function).

#### [#138](https://github.com/audiocontrol-org/deskwork/issues/138) — `repair-install.sh` cross-project leak

**Fix.** Filter the registry walk in `versions_referenced` to:
- `scope == "user"` entries always
- `scope == "project"` entries only when `projectPath == process.env.PWD` (or the cwd at script invocation)

Same filter applied to `prune_registry`.

**Files.** `scripts/repair-install.sh` (two functions).
